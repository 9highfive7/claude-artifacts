このガイドについて

このガイドでは、Claude Code の2大カスタマイズ機構 — CLAUDE.md（常時ロードのプロジェクト知識）と Skills（オンデマンドの再利用手順）— を、概念理解から実装、チーム運用まで段階的に学びます。

対象: Claude Code CLI v2.1+ を使用する開発者。前提知識として Claude Code の基本操作を理解していること。

学習目標
  • CLAUDE.md の4層階層とロード順序を説明できる
  • Skills のプログレッシブ開示（3段階）を理解する
  • CLAUDE.md と Skills を適切に使い分けられる
  • チームでの運用・Git管理戦略を設計できる
Part 1

CLAUDE.md — 概念と基礎

このセクションの学習目標
  • CLAUDE.md が何のためのファイルか説明できる
  • 「毎リクエスト読み込み」の意味とコスト影響を理解する
  • CLAUDE.md に書くべき内容と書くべきでない内容を区別できる

CLAUDE.md は、Claude Code が毎回のリクエストで自動的に読み込む Markdown ファイルです。プロジェクト固有のコーディング規約、アーキテクチャ方針、ワークフロー指示などを記述し、Claude が常にプロジェクトのコンテキストを把握した状態で作業できるようにします。

ⓘ ポイント

CLAUDE.md は「プロジェクトの常識」を伝えるファイルです。新しいチームメンバーに最初に渡すオンボーディングドキュメントに近い役割を果たします。

書くべき内容

カテゴリ例
コーディング規約「TypeScript strict mode を使用」「関数コンポーネントのみ」
アーキテクチャ「src/api/ は REST エンドポイント」「DDD のレイヤー構成」
ワークフロー「PR作成前にlintとテストを実行」「コミットメッセージはConventional Commits」
プロジェクト固有の制約「Node.js 20 以上」「PostgreSQL 16」

書くべきでない内容

NG理由代替手段
APIキーやシークレットセキュリティリスク環境変数、.env
長大なAPIリファレンストークン浪費@path でインポート分離
特定タスクの手順書毎回ロードは無駄Skills（オンデマンド）
頻繁に変わる情報メンテナンス負荷動的コンテキスト注入
⚠ コスト意識

CLAUDE.md は毎リクエストでコンテキストに読み込まれます。5,000トークンの CLAUDE.md は、50回のメッセージ交換で250,000トークンの入力コストになります。Opus 4 の場合、これだけで約 $1.25 のコストです。

理解度チェック
CLAUDE.md はいつ読み込まれますか？
Claude Code への毎回のリクエストで自動的に読み込まれます。会話の最初だけでなく、メッセージを送るたびにコンテキストに含まれます。
デプロイ手順書は CLAUDE.md に書くべきですか？
いいえ。デプロイ手順のような特定タスクの手順書は Skills として作成すべきです。CLAUDE.md に書くと、デプロイしない時も毎回トークンを消費します。
CLAUDE.md に API キーを記載してはいけない理由は？
CLAUDE.md はチームで共有されることが多く、Git にコミットされるため、シークレット情報の漏洩リスクがあります。APIキーは環境変数や .env ファイルで管理しましょう。

階層構造とロード順序

このセクションの学習目標
  • 4層の CLAUDE.md 階層を上から下まで列挙できる
  • ディレクトリツリーのウォーク動作を理解する
  • サブディレクトリの CLAUDE.md がオンデマンドロードであることを知る

CLAUDE.md は4つの階層で構成され、すべてが連結（concatenate）されてコンテキストに入ります。上書きではありません。

CLAUDE.md 階層構造（広いスコープ → 狭いスコープ）
1
Managed Policy
Linux: /etc/claude-code/CLAUDE.md
macOS: /Library/Application Support/ClaudeCode/CLAUDE.md
Windows: C:\Program Files\ClaudeCode\CLAUDE.md
2
User
~/.claude/CLAUDE.md
3
Project
./CLAUDE.md（または ./.claude/CLAUDE.md）
4
Local
./CLAUDE.local.md
レイヤースコープGit管理典型的な用途
Managed Policy組織全体IT/MDM配布セキュリティポリシー、全社コーディング規約
User個人・全PJ共通✗個人の好み（出力スタイル、使用言語）
Projectプロジェクト全体✓プロジェクト規約、アーキテクチャ
Local個人・当PJのみ✗（.gitignore）個人のローカルパス、実験設定

ロード順序の詳細

Claude Code は起動時に以下の順序でファイルを読み込み、すべて連結します。

1. Managed Policy CLAUDE.md（システムパス）
2. managed-settings.json の claudeMd キー（存在すれば）
3. ~/.claude/CLAUDE.md（ユーザー）
4. ~/.claude/rules/*.md（ユーザールール、paths なし）
5. 祖先ディレクトリの CLAUDE.md（ルート → CWD の順）
6. 各レベルの CLAUDE.local.md（同レベルの CLAUDE.md の後）
7. .claude/rules/*.md（プロジェクトルール、paths なし）
8. パススコープルール（paths あり → ファイル操作時にオンデマンド）
💡 ディレクトリウォーク

CWD が foo/bar/ の場合、foo/CLAUDE.md → foo/bar/CLAUDE.md の順に読み込まれます。サブディレクトリ（foo/bar/baz/CLAUDE.md）は起動時には読まれず、Claude がそのディレクトリのファイルを操作する時にオンデマンドで読み込まれます。

🚨 よくある落とし穴

コンパクション後の挙動: プロジェクトルートの CLAUDE.md はコンパクション後に再注入されますが、サブディレクトリの CLAUDE.md は自動再注入されません。次にそのサブディレクトリのファイルを操作した時に再ロードされます。

理解度チェック
CLAUDE.md の4層はお互いを上書きしますか？
いいえ。4層すべてが連結（concatenate）されます。上位層が下位層を上書きするのではなく、すべての内容がコンテキストに含まれます。
CLAUDE.local.md を .gitignore に入れるべき理由は？
CLAUDE.local.md は個人のローカル設定（独自のパス、実験的設定など）を記載するためのファイルです。チーム全体で共有する必要がないため、.gitignore に追加して各メンバーが自分の設定を持てるようにします。
サブディレクトリの CLAUDE.md はいつ読み込まれますか？
Claude Code の起動時には読み込まれず、Claude がそのサブディレクトリ内のファイルを実際に操作する（読み込む・編集する）時にオンデマンドで読み込まれます。

書き方とベストプラクティス

このセクションの学習目標
  • 効果的な CLAUDE.md の書き方パターンを実践できる
  • サイズ・命令数の推奨上限を知る
  • Before/After で改善パターンを理解する

サイズの目安

指標推奨値理由
行数200行以下トークンコストとのバランス
命令数~150命令以下150-200命令を超えると遵守率が低下
コンテキスト充填率30%以下余裕がないと推論品質が低下

書き方のコツ

1. 具体的な指示を書く
❌ Before

きれいなコードを書いてください。

✅ After

関数は30行以内。ESLint airbnb ルールに準拠。型定義は必ず明示。

2. 大きなドキュメントは分離する
❌ Before（1,000行）
# プロジェクト規約
...（200行のコーディング規約）...
...（300行のAPIリファレンス）...
...（500行のデプロイ手順）...
✅ After（80行）
# プロジェクト規約
@docs/coding-standards.md
@docs/architecture.md

## 重要ルール
- TypeScript strict mode 必須
- テストカバレッジ 80% 以上
...
3. HTMLコメントで補足情報を非表示にする

ブロックレベルの HTML コメント（<!-- ... -->）は CLAUDE.md 読み込み時に除去されるため、人間向けの補足メモを書いてもトークンを消費しません。

<!-- このルールは 2026-03 の技術検討会で決定。変更する場合は #arch チャンネルで議論 -->
- フロントエンドは React 18 + Next.js 15 を使用

テンプレート例

CLAUDE.md

## 技術スタック
- Language: TypeScript 5.5+（strict mode）
- Runtime: Node.js 22 LTS
- Framework: Next.js 15 (App Router)
- DB: PostgreSQL 16 + Drizzle ORM

## Git ワークフロー
- ブランチ: feature/*, fix/*, chore/*

## ディレクトリ構成
- src/app/ — ページ・ルート
- src/components/ — 共有コンポーネント
- src/lib/ — ユーティリティ・ヘルパー
- src/server/ — サーバーサイドロジック

## 追加情報
@docs/api-conventions.md
理解度チェック
CLAUDE.md の推奨行数はどれくらいですか？
200行以下が推奨です。大きなドキュメントは @path インポートで分離し、CLAUDE.md 本体は簡潔に保ちましょう。
HTML コメントを CLAUDE.md に書くとトークンを消費しますか？
いいえ。ブロックレベルの HTML コメント（<!-- -->）はコンテキスト注入前に除去されるため、トークンを消費しません。ただし、コードブロック内のコメントは保持されます。
命令数の上限はどれくらいですか？
明確なハードリミットはありませんが、約150〜200命令を超えると遵守率が低下する傾向があります。重要な命令を優先し、簡潔に記述しましょう。

自動メモリと MEMORY.md

このセクションの学習目標
  • 自動メモリの仕組みと格納場所を理解する
  • MEMORY.md の制限（200行/25KB）を知る
  • メモリタイプの4種類を把握する

Claude Code は会話中に学んだ重要情報を自動的にファイルに保存し、将来の会話で活用します。これが自動メモリ機能です。

自動メモリのディレクトリ構造
~/.claude/projects/<project>/memory/ ├── MEMORY.md ← インデックス（毎セッション読み込み） ├── user_role.md ← ユーザー情報 ├── feedback_testing.md ← フィードバック ├── project_goals.md ← プロジェクト情報 └── reference_links.md ← 参考リンク
項目値
格納場所~/.claude/projects/<project>/memory/
MEMORY.md 読み込み上限先頭200行 または 25KB（先に到達した方）
個別メモリファイル必要時にオンデマンド読み込み
手動管理/memory コマンド
無効化設定 autoMemoryEnabled: false

メモリタイプ

タイプ用途例
userユーザーの役割・好み「データサイエンティスト、Python主体」
feedbackアプローチへの修正指示「テストではDBモックを使わない」
project進行中の目標・状況「3/5以降マージフリーズ」
reference外部リソースの場所「バグはLinear INGEST で管理」
🚨 注意

重要な指示を MEMORY.md だけに頼らないでください。MEMORY.md は先頭200行の制限があり、自動生成されるため内容が不安定です。プロジェクトのルールは必ず CLAUDE.md に書きましょう。

理解度チェック
MEMORY.md は毎セッションでどれだけ読み込まれますか？
先頭200行または25KBのうち、先に到達した方までが読み込まれます。それ以降の内容はセッション開始時には読み込まれません。
4種類のメモリタイプをすべて挙げてください。
user（ユーザー情報）、feedback（フィードバック）、project（プロジェクト情報）、reference（参照リソース）の4種類です。
自動メモリは複数マシンで共有されますか？
いいえ。自動メモリはマシンローカルです。異なるマシン間で共有されません。同一リポジトリの別ワークツリーでは共有されます。

パススコープルール

このセクションの学習目標
  • パススコープルールの仕組みと配置場所を理解する
  • YAML フロントマターの paths フィールドを正しく書ける
  • グローバルルールとパス限定ルールの違いを知る

.claude/rules/ ディレクトリに Markdown ファイルを配置すると、CLAUDE.md を分割して管理できます。YAML フロントマターで paths を指定すると、特定のファイルパターンにマッチする時だけルールが適用されます。

.claude/rules/ ディレクトリ構造
.claude/ ├── CLAUDE.md ├── rules/ │ ├── code-style.md ← paths なし：常時ロード │ ├── testing.md ← paths なし：常時ロード │ ├── frontend.md ← paths あり：マッチ時のみ │ ├── api.md ← paths あり：マッチ時のみ │ └── frontend/ │ └── react.md ← サブディレクトリもOK └── settings.json

パススコープルールの書き方

frontend.md
---
paths:
  - "src/frontend/**/*.tsx"
  - "src/frontend/**/*.ts"
---

# フロントエンドルール

- React 18 の関数コンポーネントのみ使用
- CSS は Tailwind CSS v4 のユーティリティクラスを使用
- コンポーネントは src/components/ に配置
- カスタムフックは src/hooks/ に配置
api.md
---
paths:
  - "src/api/**/*.ts"
  - "src/server/**/*.ts"
---

# API 開発ルール

- 全エンドポイントに入力バリデーションを実装
- エラーレスポンスは標準フォーマットを使用
- レスポンス型は Zod スキーマで定義

Glob パターン例

パターンマッチ対象
**/*.ts全ディレクトリの TypeScript ファイル
src/**/*src/ 配下の全ファイル
*.mdルートの Markdown のみ
src/components/*.tsxcomponents 直下の TSX のみ
src/**/*.{ts,tsx}src 配下の TS/TSX（ブレース展開）
💡 ユーザーレベルのルール

~/.claude/rules/ に配置すると、全プロジェクトに適用されるルールを定義できます。プロジェクトルールより先に読み込まれます。

理解度チェック
paths フロントマターがないルールファイルはいつ読み込まれますか？
paths がないルールファイルは CLAUDE.md と同じタイミングで常時ロードされます。パス条件なしのグローバルルールとして扱われます。
パススコープルールのメリットは何ですか？
特定のファイルパターンにマッチした時だけルールが読み込まれるため、トークンの節約になります。フロントエンドのルールをバックエンド作業中に読み込む無駄がなくなります。
.claude/rules/ のサブディレクトリにファイルを置けますか？
はい。再帰的に探索されるため、.claude/rules/frontend/react.md のようなサブディレクトリ構成も有効です。

インポート構文

このセクションの学習目標
  • @path インポート構文を使えるようになる
  • 再帰インポートの上限（5ホップ）を知る
  • 大きな CLAUDE.md を分割する設計パターンを理解する

CLAUDE.md 内で @path/to/file と記述すると、指定ファイルの内容がインラインで展開されます。

項目仕様
構文@path/to/file
相対パスインポート元ファイルからの相対
絶対パス使用可能
~/ プレフィックス使用可能（@~/.claude/shared.md）
再帰インポートインポート先からさらにインポート可（最大5ホップ）
初回承認外部インポートの初回は承認ダイアログが表示される
CLAUDE.md（分割パターン）
# プロジェクト名

## 概要
@README.md

## コーディング規約
@docs/coding-standards.md

## API設計ガイド
@docs/api-conventions.md

## 個人設定
@~/.claude/my-project-instructions.md
⚠ トークンへの影響

インポートされたファイルは起動時に展開されてコンテキストに読み込まれます。インポート自体がトークンを削減するわけではありません。大きなファイルをインポートすれば、それだけコンテキストを消費します。インポートの目的はあくまで管理の分離です。

理解度チェック
@path インポートはトークンを節約しますか？
いいえ。インポートされたファイルは展開されてコンテキストに含まれるため、トークン消費量は同じです。インポートのメリットは管理の分離（メンテナンス性の向上）にあります。
インポートの相対パスは何を基準にしますか？
@path の相対パスは、CWD（作業ディレクトリ）ではなく、インポート元のファイルが存在するディレクトリを基準に解決されます。
再帰インポートの上限は？
最大 5ホップ です。A が B をインポートし、B が C をインポートし…と続く場合、5段階目まで展開されます。
Part 2

Skills — 概念と基礎

このセクションの学習目標
  • Skills が CLAUDE.md と何が異なるか説明できる
  • Skills のオンデマンドロードの仕組みを理解する
  • Skills の配置場所とディレクトリ構造を知る

Skills（Agent Skills）は、特定のタスクを実行するための再利用可能な手順書です。CLAUDE.md が「常に知っておくべき知識」であるのに対し、Skills は「必要な時だけ呼び出す手順」です。

CLAUDE.md

常時ロード: 毎リクエストでコンテキストに含まれる。プロジェクトの「常識」を伝える。

Skills

オンデマンド: 呼び出された時だけコンテキストに入る。特定タスクの「手順」を定義する。

ディレクトリ構造

Skills のディレクトリ構造
.claude/skills/ ├── deploy/ │ ├── SKILL.md ← エントリポイント（必須） │ ├── scripts/ │ │ └── validate.sh ← 実行スクリプト │ └── references/ │ └── REFERENCE.md ← 詳細ドキュメント ├── code-review/ │ └── SKILL.md ├── test-runner/ │ └── SKILL.md └── SKILL.md ← ルートレベルも可

配置スコープ

スコープパス適用範囲
Enterpriseマネージド設定組織全体
Personal~/.claude/skills/<name>/SKILL.md全プロジェクト
Project.claude/skills/<name>/SKILL.md当プロジェクト
Plugin<plugin>/skills/<name>/SKILL.mdプラグイン利用先
ⓘ レガシー互換

.claude/commands/ ディレクトリも引き続きサポートされています。同名のスキルとコマンドがある場合、スキルが優先されます。

理解度チェック
Skills と CLAUDE.md の最大の違いは何ですか？
ロードタイミングです。CLAUDE.md は毎リクエストで常時ロードされますが、Skills は呼び出された時だけオンデマンドでロードされます。
SKILL.md は必ずサブディレクトリに配置する必要がありますか？
いいえ。.claude/skills/SKILL.md のようにルートレベルに直接配置することも可能です。ただし、補助ファイル（スクリプト、リファレンス）がある場合はサブディレクトリ構成が推奨されます。
個人用スキルを全プロジェクトで使うにはどこに配置しますか？
~/.claude/skills/<skill-name>/SKILL.md に配置します。ユーザーレベルのスキルは全プロジェクトで利用可能です。

プログレッシブ開示

このセクションの学習目標
  • 3段階のプログレッシブ開示モデルを図示できる
  • 各レベルのトークンコストを理解する
  • 自動コンパクション時のスキル保持ルールを知る

Skills は3段階のプログレッシブ開示モデルを採用しています。必要な情報だけを段階的にコンテキストに入れることで、トークン消費を最小限に抑えます。

プログレッシブ開示の3段階
L1
メタデータ（~100 tokens / skill）
name + description → 常時ロード
L2
指示本文（可変）

L3

💡 トークン節約効果

10個のスキルがある場合、Level 1 のコストは約 1,000トークン（100 × 10）です。全スキルを CLAUDE.md に埋め込んだ場合と比較して、呼ばれないスキルのトークンがゼロになるため、約30%のトークン節約が実現できます。

コンテキストが長くなると、スキルの内容は自動的にコンパクションされます。

4. デプロイ後の検証

理解度チェック
SKILL.md の最低限必要な要素は何ですか？

disable-model-invocation: true はどんな場面で使いますか？

user-invocable: false はどんな用途ですか？
/ メニューに表示せず、Claude のみが自動で呼び出せるスキルになります。背景知識として Claude に判断材料を与えたいが、ユーザーが直接実行する必要がない場合に使います。

理解度チェック
allowed-tools フィールドの目的は何ですか？

context: fork と agent: Explore を組み合わせるとどうなりますか？
Explore タイプのサブエージェントが起動し、SKILL.md の内容をプロンプトとして受け取ります。Explore エージェントは CLAUDE.md をスキップし、読み取り専用のコード探索に特化した動作をします。
description と when_to_use の文字数上限は？
両フィールドの合計で 1,536文字 です。Level 1 メタデータとしてコンテキストに含まれるため、簡潔に書くことが重要です。

動的コンテキスト注入

!`command` 構文を使うと、スキルがロードされる前にシェルコマンドを実行し、結果を Markdown 内に展開できます。

インライン形式
---
description: 未コミットの変更を要約してリスクを指摘する
---

## 現在の変更内容
!`git diff HEAD`

## 最近のコミット
!`git log --oneline -5`

## 手順
上記の変更を2〜3文で要約し、リスクを指摘してください。
複数行形式
## 環境情報
```!
node --version
npm --version
git status --short
```
⚠ 前処理タイミング

動的コンテキスト注入は前処理です。Claude が見る前にコマンドが実行され、結果に置換されます。出力は再スキャンされず、出力内の !`...` は実行されません。

🚨 構文の注意点

! は行頭またはスペースの直後にある場合のみコマンドとして認識されます。KEY=!`cmd` のように他の文字の直後にある場合はリテラルテキストとして扱われ、コマンドは実行されません。

理解度チェック
$ARGUMENTS がスキル本文中に存在しない場合はどうなりますか？
引数は自動的に本文末尾に ARGUMENTS: <value> として自動付加されます。明示的に $ARGUMENTS を書いておく方が、配置を制御できるため推奨です。
動的コンテキスト注入 !`command` はいつ実行されますか？
スキルがコンテキストに読み込まれる前（前処理段階）に実行されます。Claude はコマンドの出力結果だけを見ます。
${CLAUDE_SKILL_DIR} はどんな用途に使いますか？
スキルに同梱されたスクリプトやリソースファイルを参照する場合に使います。例: python3 ${CLAUDE_SKILL_DIR}/scripts/analyze.py

よくある間違いと対策

1. CLAUDE.md の肥大化

CLAUDE.md が 1,000行を超えると、トークンコストが増大し、命令の遵守率も低下します。

対策: @path インポートで分離、パススコープルールの活用、手順書は Skills に移行。

2. CLAUDE.md と Skills の重複

同じルールを CLAUDE.md とスキルの両方に書くと、矛盾が生じやすくなります。

対策: 事実はCLAUDE.md、手順はSkills。一か所に書くことを原則にする。

3. パス未指定のルール乱用

フロントエンド限定のルールをグローバルに置くと、バックエンド作業時にも無駄にロードされます。

対策: .claude/rules/ に paths 付きで配置する。

4. 毎回必要な情報をスキルに入れる

常に参照すべき情報をスキルに入れると、呼び出さない限りClaudeが認識しません。

対策: 毎回必要な情報は CLAUDE.md に書く。

5. MEMORY.md への重要指示の依存

MEMORY.md は先頭200行制限があり、自動生成で内容が不安定です。

対策: プロジェクトルールは必ず CLAUDE.md に記載。メモリは補助的な参考情報のみ。

6. 曖昧な指示

「きれいなコードを書け」は解釈が定まりません。

対策: 具体的な基準を書く。「関数30行以内」「変数名はcamelCase」「エラーは Result 型で返す」

用語集

CLAUDE.md
Claude Code が毎リクエストで自動読み込みする Markdown ファイル。プロジェクト固有のルール・知識を伝える。4層の階層構造（Managed Policy → User → Project → Local）を持つ。
Skills（Agent Skills）
必要時にオンデマンドでロードされる再利用可能な手順書。.claude/skills/ に SKILL.md として配置。プログレッシブ開示で3段階に情報を展開する。
プログレッシブ開示
情報を段階的に開示する設計パターン。Skills では Level 1（メタデータ、~100 tok）→ Level 2（指示本文）→ Level 3（リソース）の3段階でトークン消費を最小化する。
フロントマター（Frontmatter）
SKILL.md やルールファイルの先頭に --- で囲んで記述する YAML メタデータ。name, description, arguments, allowed-tools 等のフィールドでスキルの挙動を制御する。
パススコープルール
.claude/rules/ に配置し、YAML フロントマターの paths フィールドで glob パターンを指定したルール。マッチするファイルを操作する時だけロードされる。
動的コンテキスト注入
SKILL.md 内の !`command` 構文でシェルコマンドを前処理実行し、結果を Markdown に展開してから Claude に渡す機能。git diff, 環境情報等の動的データの取り込みに使う。
コンパクション（Compaction）
コンテキストウィンドウが上限に近づいた時に、会話やスキルの内容を要約圧縮する処理。/compact で手動実行も可能。スキルは先頭5,000トークンが保持される。
Managed Policy
IT管理者が組織全体に配布する CLAUDE.md / 設定。最も優先度が高く、ユーザーやプロジェクトの設定で上書きできない。MDM やシステムパスで配置。
自動メモリ（Auto Memory）
Claude Code が会話中に学んだ情報を自動的にファイルに保存する機能。~/.claude/projects/<project>/memory/ に格納。MEMORY.md は先頭200行/25KBが毎セッションで読み込まれる。
AGENTS.md
複数のAIツール（Claude Code, Cursor, GitHub Copilot 等）が共通で参照する指示ファイルのクロスツール標準。symlink や @AGENTS.md インポートで Claude Code と互換性を確保できる。
