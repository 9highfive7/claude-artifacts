**第1部: CLAUDE.md 編**

🎯 本セクションの学習目標

CLAUDE.mdが果たす役割、ロードされる仕組み、そしてモデルに命令を正しく遵守させるための「引き算の書き方」を習得します。

1. CLAUDE.mdとは何か

Claude Codeにおいて、**CLAUDE.md** は「エージェントにプロジェクト特有のコンテキスト（不変のルール・常識）を与えるための静的な長期記憶ファイル」です。 毎セッションの立ち上げ時に、Claudeは自動的にこのファイルを走査して読み込み、自分がどのプロジェクトで、どのような規約に従って行動すべきかを理解します。

ℹ️ なぜ必要なのか？何を解決するのか？
毎回のチャットで「このプロジェクトはPythonで動いていて、インデントはスペース4つで、ビルドコマンドは `make build` です」と説明し直す必要をなくします。開発者の認知的負荷を激減させる、プロジェクト専用のオンボーディング・資料の役割を果たします。

2. 配置場所と多層的なロード階層

CLAUDE.mdは単一のファイルに留まりません。カレントディレクトリからファイルシステムのルート（/）までを遡る形で、異なるスコープのルールを検出し、結合して読み込みます。

適用範囲 (スコープ) 配置場所 主な用途 共有範囲
組織共通ルール /etc/claude-code/CLAUDE.md 企業のコーディング標準、セキュリティポリシー 社内全員 (マシン共有)
ユーザー個人設定 ~/.claude/CLAUDE.md 個人開発者の書き方の好み、よく使う短縮コマンド 自分自身 (全プロジェクト)
プロジェクト規約 ./CLAUDE.md または ./.claude/CLAUDE.md プロジェクト固有のビルド/検証、アーキテクチャ方針 Gitを介してチーム全員で共有
個人ローカル設定 ./CLAUDE.local.md Git追跡対象外にする個人的なローカル検証用設定 自分専用 (現在のプロジェクト)
⚠️ ローカル環境設定の注意
CLAUDE.local.md は個人の認証情報やサンドボックスURLを格納するのに適していますが、誤ってリポジトリにコミットしないよう、必ず .gitignore に追加してください。

3. 書き方の基本と推奨構成

CLAUDE.mdはプレーンなMarkdownで記述します。ロードされた内容は常にコンテキストウィンドウを消費するため、**「200行以下」**で簡潔に保つ必要があります。

推奨セクション構成:

  • **Build & Test:** 実際に動作する正確なコマンド。
  • **Coding Standards:** インデント数、エラー処理、命名規則。
  • **Architecture:** 主要技術スタック、ディレクトリ構成。
  • **Imports:** `@path/to/file` の構文を使用し、READMEやpackage.json、その他個別のルール（.claude/rules/）をインポート可能。

4. Before & After で見る記述例

❌ 悪い例 (抽象的で冗長)

「コードはきれいに書いてください。テストもちゃんと走らせること。フロントエンドはReactを使っていますが、色々古いコンポーネントもあります。エラーは適切にログに出力するように心がけてください。」

👍 良い例 (具体的・機械可読)
# Build & Test commands
- Build: `npm run build`
- Test: `npm test`

# Code Style
- Use 2-space indentation
- Use Functional Components in React
- Error handling: Always wrap in try/catch and log to `logger.error`

5. 便利な機能：対話中の「#」プレフィックス

セッション中に # キーを押してインストラクションを入力すると、Claude Codeは対話中での開発者の修正やメモを検知し、該当する `CLAUDE.md` ファイルに自動的にその「記憶」をマージ・追記します。これにより、マニュアルで編集する手間を省きながらメモリを最新に維持できます。

🧩 理解度チェック：CLAUDE.md
Q1. CLAUDE.mdは1ファイルあたり何行以内に抑えるべきですか？

A1. 200行未満。 それ以上長いファイルは、Claudeの命令遵守率を下げ、無用なトークンを毎ターン消費させてしまいます。

Q2. プロジェクト内でチーム共通ルールと自分専用のローカル環境定義を使い分けたい場合、どうすればいいですか？

A2. 共通ルールは CLAUDE.md に記述してGit管理し、個人設定は CLAUDE.local.md に書いて .gitignore に追加します。

**第2部: Skills (Agent Skills) 編**

🎯 本セクションの学習目標

Agent Skillsのメンタルモデルを確立し、YAMLフロントマターを用いた設計手順と、事前評価式のシェルコマンド埋め込み（事前処理）の実装方法を習得します。

1. Skillsとは何か

**Agent Skills**は、特定の手続きや複雑なタスクを、Claudeが「必要な時にだけ」引き出して実行できる機能です。 常時脳内にロードされているCLAUDE.mdとは異なり、**必要なときだけ読み込まれるため、大幅なトークンの節約**になります。

🧠 オンデマンドロードの仕組み (Progressive Disclosure)

Claudeは起動時にすべてのSkillの「description」のみを薄く記憶します。

📂 description で適合判定 ➡️ 🎯 特定Skillの中身全体をロード

2. 構造とファイル形式

Skillは必ず SKILL.md というファイルを含む、独立したフォルダで構成されます。 

my-skill/
 ├── SKILL.md       # メインの指示とフロントマター（必須）
 ├── template.md    # 補助的なテンプレート（オプション）
 └── scripts/
     └── helper.py  # Claudeが呼び出す補助スクリプト（オプション）

3. SKILL.md のフロントマター

YAML形式で動作パラメーターを指定します。

---
name: create-pr
description: Gitのステージング領域から綺麗なプルリクエストを生成します
disable-model-invocation: true
allowed-tools:
  - Bash
---
ここに、Claudeに実行させたい詳細な手順を記載します。

4. 高度な実装パターン：動的なコンテキスト注入

`` !`command` `` 構文を使うことで、ClaudeがこのSkillを読み込む**直前**に、ホストマシンのシェルで事前コマンドを実行し、その実行結果（stdout）を自動的にプロンプトへ埋め込むことができます。 

# システムの稼働状況
```!
uptime
uname -a
```
🔥 組織導入時のセキュリティリスク
誰が書いたか分からない不正なリポジトリを不用意にローカルで開くと、Skillsの読み込み時に悪意のあるコマンドが事前実行される（RCE）リスクがあります。組織展開の際は、共有設定ファイル等で "disableSkillShellExecution": true を指定し、この評価型コマンド実行を無効化することを推奨します。
🧩 理解度チェック：Agent Skills
Q1. CLAUDE.mdとSkillsの最も大きな違いは何ですか？

A1. ロードのタイミングとトークン消費。 CLAUDE.mdは常時ロードされるため継続的にトークンを消費しますが、Skillsは該当タスクが発生した時にオンデマンドでロードされるため無駄なトークンを消費しません。

Q2. Skill内の markdown に !「コマンド」 と書いた場合、いつ実行されますか？

A2. ClaudeがSkillを読み込む（呼び出される）直前に、開発者のローカル環境（ホスト）上で実行されます。 実行結果の文字列が、プロンプトの一部に変換されてからClaudeに渡されます。

**第3部: 比較と使い分け**

🎯 本セクションの学習目標

Claude Codeで利用可能な複数の文脈制御機能（CLAUDE.md、Skills、サブエージェント、スラッシュコマンド）を状況に応じて最適に使い分けられるようにします。

ツール ロード条件 最適ユースケース ファイル実体
CLAUDE.md 常時自動 常に適用すべき基本的な命名規則、ビルド・テスト環境の定義 CLAUDE.md
Skills 必要に応じて（自動/手動） 定形手順（デプロイ、PR作成、定型コードビジュアル生成など） .claude/skills/*/SKILL.md
サブエージェント ユーザーが指示した時 大きなコードリファクタリング、広範囲なコード影響度調査の分離 .claude/agents/*.md
スラッシュコマンド ユーザーによる手動のみ 環境管理、コンテキスト圧縮（/compact）、モデル切替 組み込み (またはカスタム)

使い分け決定ツリー

  • **「すべてのチャットで、Claudeにビルド方法を知っていてほしい」** ➡️ **CLAUDE.md**
  • **「特定の言語や特定のディレクトリだけで適用したいルールがある」** ➡️ **.claude/rules/** の path-specific ルール
  • **「普段は忘れていてほしいが、週に1回のデプロイ時だけ長文マニュアルを参照してほしい」** ➡️ **Agent Skills**

**第4部: 運用とチーム展開**

🎯 本セクションの学習目標

チームでClaude Codeを導入する際のGit運用ポリシー、レビューサイクル、および機密情報の安全な取り扱い方を学びます。

1. Git管理のゴールデンルール

  • **コミットすべきもの:** CLAUDE.md、.claude/rules/、.claude/skills/。これらはチーム全体で共通の「AI協働用ドキュメント」として常にバージョン管理下に置きます。
  • **Git無視（.gitignore）すべきもの:** CLAUDE.local.md、.claude/settings.local.json。

2. チーム内での標準化プロセス

チームの誰かがClaude Codeに「変なコードの書き方をされた」または「テストコマンドを間違えられた」場合、対話側で修正させるだけでなく、その場で CLAUDE.md のルールを1行改善（Pull Requestを作成）する文化を醸成することがAI協働をスケールさせる鍵となります。

⚠️ 機密情報の漏洩を防ぐ
CLAUDE.md や SKILL.md に直接、パスワードや秘密鍵（APIキー）を書き込んではいけません。それらは環境変数経由（例：`${MY_API_KEY}`）で参照するよう設定するか、前述の CLAUDE.local.md に追い出してください。

🛠️ 手を動かして学ぶ課題

以下の課題に実際に取り組み、設計スキルを確固たるものにしましょう。

課題1: 黄金のCLAUDE.mdを `/init` から育てる

状況: あなたはNode.js + Jest環境のWeb開発プロジェクトにClaudeを導入しようとしています。

手順:

  1. プロジェクトのルートで claude /init を実行します。
  2. 自動生成された CLAUDE.md の中身をエディタで確認します。
  3. コーディング基準セクションに「非推奨の関数（例: `var` 宣言は禁止、必ず `const` を使う）」を追記します。

課題2: コミット作成オートメーションSkill

状況: コミットメッセージに「[FE]」「[BE]」などのプレフィックス付与を強制したいとします。

実装サンプル (.claude/skills/smart-commit/SKILL.md):

---
name: smart-commit
description: プロジェクト規約に合わせた丁寧なGitコミットを作成します
disable-model-invocation: true
allowed-tools:
  - Bash
---

# 役割
以下のGitコミット規約に厳格に従ってコミットを作成してください。

## 規約
- フロントエンド関連の変更: `feat[FE]: <内容>` もしくは `fix[FE]: <内容>`
- バックエンド関連の変更: `feat[BE]: <内容>` もしくは `fix[BE]: <内容>`
- それ以外の雑多なタスク: `chore: <内容>`

## 手順
1. `git diff --cached` で現在のステージング状況を確認する。
2. 変更履歴に基づいて適切なプレフィックスを自動判定する。
3. `git commit -m "<作成したメッセージ>"` を実行する。

課題3: Python版・コード構造自動ビジュアライザ

Claudeに以下のように依頼してください。「プロジェクトを分析して可視化するスクリプトを Skills として実装してほしい」と伝えるだけで、Claudeは .claude/skills/code-visualizer/ を立ち上げ、自動的にHTMLグラフを生成するPythonコードを作成してくれます。

**用語集 (Glossary)**

Compaction (コンテキスト圧縮)
チャット履歴がモデルのコンテキスト制限に近づいた際、過去の履歴を要約し、不要なトークンを解放するClaude Codeの内製プロセス。
Progressive Disclosure (段階的開示)
必要な瞬間にのみ詳細な情報を展開し、通常時は不要な注意リソース（トークン）を削減する設計思想。Agent Skillsに採用されている。
サブエージェント (Sub-agents)
メインの対話とは完全に別の独立したシステム文脈をフォークして並行起動し、特定の重たい作業を完全に裏で並列処理させる仕組み。
