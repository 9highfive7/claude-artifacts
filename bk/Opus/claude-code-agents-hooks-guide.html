第0部 全体像と用語整理

🎯 学習目標
  • 「Agent」「SubAgent」「Hooks」「Skills」「Slash Commands」「MCP」の責務の違いを説明できる
  • Claude Code自体が1つのAgentである、という前提を理解する
  • 「決定論的」と「LLM判断」の違いを区別できる

そもそも「Agent」とは

本ガイドにおけるAgentとは、「LLM（大規模言語モデル）がツールを自律的に使い、ループを回しながらタスクを遂行する仕組み」の総称です。Claude Code自体が、ファイル読み書き・Bash実行・Web検索などのツールを備えた一つのAgentとして動作しています。

つまり、Claude Codeを起動した瞬間にあなたはすでに1つのAgentと対話している状態です。本ガイドで扱うSubAgentやHooksは、この親Agentを拡張・制御するための機構です。

5つの拡張機構の関係

Claude Codeには複数の拡張機構があり、それぞれ役割が異なります。混同しないことが習熟の第一歩です。

機構責務トリガー確実性
SubAgent独立コンテキストで専門タスクを実行LLM判断 / 明示LLM判断
Skills指示書・手順書をパッケージ化（メインコンテキストで動作）LLM判断 / 明示LLM判断
Hooksライフサイクル上で確実に処理を実行（lint, 通知, ブロック等）イベント発火決定論的
Slash Commandsユーザーが入力欄から手動で呼び出す機能ユーザー入力決定論的
MCP外部システムをツールとして接続するプロトコルLLM判断LLM判断
💡 覚え方 「LLMに判断させたいならSubAgent / Skills / MCP、毎回必ず実行させたいならHooks、ユーザー主導ならSlash Command」 — この3軸で整理できます。

本ガイドのスコープ

本ガイドでは、Claude Codeを使ったソフトウェア開発の現場でもっとも頻繁に拡張ポイントとなる3つ、つまり Agent全般・SubAgent・Hooks に絞って深掘りします。Skills / MCP / Slash Commandsは比較目的で第4部で触れます。

✅ 理解度チェック（第0部）
  1. 「Claude Code自体が1つのAgentである」とはどういう意味か？
    解答を表示
    Claude CodeはLLM（Claude）+ ツール（Read/Write/Bash等）+ 実行ループの組み合わせで動く、すなわちそれ自体が自律的にタスクを遂行する「Agent」の実装である、という意味。SubAgentやHooksはこの親Agentを補強・制御する追加機構。
  2. 「決定論的」な拡張機構はどれか？ また、それは何を意味するか？
    解答を表示
    Hooksが決定論的。「LLMがその時の判断で実行を選ぶ」のではなく、定義したイベントが発生したら必ず実行されることを意味する。lintや監査ログのような「漏れてはならない処理」に向いている。
  3. SubAgentとSkillsの本質的な違いを1文で説明せよ。
    解答を表示
    SubAgentは独立した別コンテキストでタスクを実行するのに対し、Skillsはメインの会話コンテキスト内で指示書として展開・実行される。

第1部 Agent 編 — 基礎

🎯 学習目標
  • エージェントループの構造を図で説明できる
  • Agentの構成要素（システムプロンプト / ツール / メモリ / ループ制御）を列挙できる
  • Claude Codeをスクリプトやヘッドレスで呼び出す方法を理解する

1. Agentの基本概念

Agentは「LLMがツール呼び出しを自分で選択しながら目的を達成するシステム」です。ユーザーから受け取った指示（プロンプト）を解釈し、必要なツールを使い、結果を観測しながら次の行動を決める、というサイクルを繰り返します。

2. エージェントループ

Claude Codeの内部では、以下のようなエージェントループが回っています。

このループの各ステップで、Hooksが介入できる箇所が用意されています（例: ②の前に PreToolUse が、②の後に PostToolUse が発火）。また、①の段階で「これは専門領域だ」と判断されると SubAgent に処理が委譲される、という形で各機構が組み合わさります。

3. Agentを構成する要素

要素役割Claude Codeでの実体
システムプロンプトAgentの人格・指示Claude Code内蔵 + CLAUDE.md + ユーザー設定
ツール外界と相互作用する手段Read, Write, Edit, Bash, Grep, Glob, WebFetch, Task 等
メモリセッションをまたぐ知識CLAUDE.md（プロジェクト・ユーザー・ローカル階層）/ auto memory
実行ループ制御停止条件・最大ターン数maxTurns、Stop hook、auto-compaction 等
許可制御どのツールを何処まで使えるかpermissions 設定、permissionMode

4. Agent SDK / ヘッドレス利用

Claude Codeは対話モードだけでなく、スクリプト・CI/CDから呼び出すことができます。これは「Agent SDK」と呼ばれ、Python・TypeScriptのライブラリ版と、CLIの非対話モード（claude -p）の2系統があります。

bash# 非対話モード（ヘッドレス）でClaude Codeを実行
claude -p "認証モジュールのバグを修正して" \
  --allowedTools "Read,Edit,Bash" \
  --output-format json

# --bare で起動高速化（hooks/skills/MCPの自動ロードを抑制）
claude --bare -p "このファイルを要約して" --allowedTools "Read"

# 構造化出力を要求
claude -p "このプロジェクトを要約" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"summary":{"type":"string"}}}'
📌 公式情報 ヘッドレス実行は claude-code/headless、SDK詳細は agent-sdk/overview を参照。
📝 この章の要点
Agentは「LLM + ツール + ループ」の組み合わせ。Claude Code自体がこのパターンの実装であり、SubAgentとHooksはそれぞれ「分岐」と「介入点」を提供する。スクリプトから呼び出すには claude -p またはPython/TypeScript SDKを使う。
✅ 理解度チェック（第1部）
  1. エージェントループの4ステップを順に挙げよ。
    解答を表示
    ① 思考/計画 → ② ツール呼び出し → ③ 結果の観測 → ④ 完了判定 → （未完了なら①へ戻る）
  2. Agentを構成する5要素のうち、「セッションをまたいで知識を保持する」要素は何か？
    解答を表示
    メモリ。Claude Codeでは CLAUDE.md（プロジェクト/ユーザー/ローカル階層）や auto memory がこれに該当する。
  3. claude -p と claude --bare -p の違いは？
    解答を表示
    通常の -p はhooks・skills・MCP・CLAUDE.mdなどをすべてオートロードするが、--bare はそれらを一切ロードせず、明示的に渡したフラグだけが有効になる。CI環境で再現性を確保したいときに使う。

第2部 SubAgent 編 — 詳細

🎯 学習目標
  • SubAgentが独立コンテキストを持つ意義を説明できる
  • .claude/agents/<name>.md のフロントマター必須項目を書ける
  • 自動委譲と明示呼び出しの2パターンで使い分けられる
  • 3種類の実用SubAgent（レビュー / テスト / ドキュメント）を作れる

1. SubAgentとは

SubAgent（サブエージェント）は、親であるClaude Codeから委譲される専門特化型のエージェントです。各SubAgentは独自のシステムプロンプト・ツールセット・モデル・パーミッションを持ち、独立したコンテキストウィンドウで動作します。

独立コンテキストの最大のメリットは「親のコンテキスト汚染を防ぐ」点です。例えば、何千行ものログ解析やコードベース探索を親エージェントで直接やると、すぐにコンテキストが一杯になり、本筋のタスクに支障が出ます。SubAgentに委譲すれば、SubAgent側がログを全部読んでも親にはサマリだけが返ってくるので、親のコンテキストは綺麗に保たれます。

主なメリット（公式ドキュメントより）:

  • コンテキスト保持: 探索・実装作業をメイン会話から切り離す
  • 制約の強制: SubAgentが使えるツールを限定できる
  • 再利用: ユーザーレベルのSubAgentは全プロジェクトで共有可
  • 専門化: ドメイン特化のシステムプロンプトを与えられる
  • コスト管理: HaikuなどよりCPなモデルにルーティング可能

2. ビルトインSubAgent

Claude Codeには最初から有用なSubAgentが組み込まれています。

名前モデルツール用途
ExploreHaiku（高速）読み取り専用コードベース検索・探索
Plan親と同じ読み取り専用Planモードでの調査
general-purpose親と同じ全ツール複雑な多段タスク
statusline-setupSonnet—/statusline 設定支援
claude-code-guideHaiku—Claude Code機能のQA
📌 重要な仕様 ExploreとPlanはCLAUDE.mdとgit状態をスキップします（高速化のため）。それ以外のビルトイン・カスタムSubAgentはCLAUDE.mdを読み込みます。

3. 設定ファイルの構造

カスタムSubAgentは .md ファイル（YAML frontmatter + Markdown本文）として定義します。配置場所により適用範囲が変わります。

場所スコープ優先順位
Managed settings組織全体1（最高）
--agents CLIフラグ現セッションのみ2
.claude/agents/プロジェクト3
~/.claude/agents/ユーザー（全プロジェクト）4
Plugin agents/プラグイン有効時5（最低）

最小構成

markdown---
name: code-reviewer
description: Reviews code for quality and best practices. Use proactively after code changes.
tools: Read, Glob, Grep
model: sonnet
---

You are a code reviewer. When invoked, analyze the code and provide
specific, actionable feedback on quality, security, and best practices.

Focus on:
- Naming clarity
- Error handling
- Test coverage
- Security risks (input validation, secrets in code)

主要フロントマター項目（公式仕様）

フィールド必須説明
name✅小文字とハイフンのみの一意識別子
description✅いつClaudeがこのSubAgentに委譲すべきかを記述（自動委譲のトリガー）
tools—許可するツール（カンマ区切り）。省略で親から継承
disallowedTools—禁止ツール（拒否リスト）
model—sonnet / opus / haiku / モデルID / inherit
permissionMode—default / acceptEdits / auto / dontAsk / bypassPermissions / plan
maxTurns—最大エージェントターン数
skills—起動時にプリロードするSkill
mcpServers—このSubAgent専用MCPサーバ
hooks—このSubAgentのライフサイクルhook
memory—user / project / local（永続メモリ）
background—trueで背景実行
isolation—worktreeで一時git worktreeを使う
color—UIでの色（blue, orange等）
⚠️ descriptionの書き方が肝 Claudeは description を読んで「今このタスクにこのSubAgentを使うべきか」を判断します。「Use proactively after ...」「Specialist for ...」のようにトリガー条件を具体的に書きましょう。曖昧な説明は呼ばれません。

4. 動作の仕組み

(a) 自動委譲

ユーザーが何も指示しなくても、Claudeが状況と description を見て自動的にSubAgentに委譲します。

(b) 明示呼び出し（3パターン）

text# 1. 自然言語で名前を指定
Use the test-runner subagent to fix failing tests

# 2. @-mention（確実にそのSubAgentを呼ぶ）
@code-reviewer please look at the auth changes

# 3. セッション全体をそのSubAgentで実行
claude --agent code-reviewer

(c) 起動時に何がロードされるか（公式仕様）

非フォーク型のSubAgentが起動するとき、コンテキストには以下が含まれます。

  • そのSubAgent自身のシステムプロンプト
  • 親が書く委譲メッセージ（タスクの概要）
  • CLAUDE.md 階層全体（Explore / Plan は除く）
  • git状態スナップショット（Explore / Plan は除く）
  • skills フィールドで指定されたSkillのフル内容

つまり親の会話履歴は引き継がれません。これがコンテキスト分離の本質です（例外: /fork による「フォーク」は親履歴を引き継ぐ）。

5. ハンズオン: 3つの実用SubAgentを作る

実例① コードレビュー専門

markdown---
name: code-reviewer
description: Senior code reviewer. Use proactively immediately after the user makes code changes, to surface bugs, security risks, and style issues.
tools: Read, Grep, Glob, Bash
model: sonnet
color: orange
---

You are a senior code reviewer. When invoked:

1. Run `git diff` to identify the changed files
2. Read each changed file fully
3. Provide a structured review with:
   - Critical issues (must fix)
   - Suggestions (should consider)
   - Nitpicks (optional)
4. For each issue, cite file path and line number

Focus on: correctness, error handling, security (input validation, secret leaks),
readability, and test coverage.

実例② テスト生成専門

markdown---
name: test-writer
description: Generates unit tests for new or modified functions. Invoke after implementing a feature, before merging.
tools: Read, Write, Edit, Grep, Glob, Bash
model: sonnet
color: green
---

You are a testing specialist. For each target file:

1. Detect the project's test framework (look for jest, pytest, vitest, etc.)
2. Identify functions lacking tests using grep / glob
3. Write tests covering: happy path, edge cases, error cases
4. Place tests following the project's existing convention
5. Run the test suite and verify they pass

Do not modify production code. If a function is untestable as-is,
report what would need to change instead of making the change yourself.

実例③ ドキュメント執筆専門

markdown---
name: doc-writer
description: Writes and updates README, API docs, and inline JSDoc/docstrings. Use when documentation is missing or out of date.
tools: Read, Write, Edit, Grep, Glob
model: haiku
color: blue
---

You are a technical writer. Produce documentation that is:

- Accurate (verify against the actual code, never hallucinate API signatures)
- Concise (no marketing language)
- Example-driven (every non-trivial function gets a usage example)
- Consistent with the project's existing tone

Always read the file you're documenting first. Match heading levels, code-fence
language tags, and link styles already used in the repo.

動作確認の手順

  1. プロジェクトルートで mkdir -p .claude/agents
  2. 上記いずれかのファイルを .claude/agents/code-reviewer.md として保存
  3. Claude Codeを起動して /agents コマンドで一覧表示
  4. @code-reviewer look at my recent changes と入力して呼び出し
  5. 呼び出されない場合は description のトリガー語句を強化（「Use proactively」等）

6. ベストプラクティス

  • 1責務1SubAgent: 「コードレビュー＋テスト＋デプロイ」を1つにまとめない
  • tools は最小限: コードレビューにWriteは不要。攻撃面を減らし、トークンも節約
  • description は具体的に: 「いつ呼ばれるべきか」を明示。曖昧だと呼ばれない
  • モデルは適材適所: 軽量な探索はHaiku、複雑な判断はSonnet/Opus
  • 並列実行を活用: 独立した調査は複数SubAgentで並列に
  • memoryは慎重に: 学習を蓄積させたいときだけ。乱用すると古い知見が悪さをする

✅ 理解度チェック（第2部）
  1. SubAgentが「独立コンテキスト」を持つことの実用的なメリットを1つ挙げよ。
    解答を表示
    大量のログ・検索結果などを処理しても、親のコンテキストが汚染されない。結果として親は本来のタスク（実装や設計判断）に必要な情報を保持できる。トークンコストも節約できる。

第3部 Hooks 編 — 詳細

🎯 学習目標
  • Hooksが「決定論的」である意味を説明できる
  • 主要なHookイベント（PreToolUse / PostToolUse / UserPromptSubmit / Stop / Notification 等）を区別できる
  • settings.json にHookを設定し、終了コードで挙動を制御できる
  • セキュリティリスクを理解し、安全な運用ができる

1. Hooksとは

Hooksは、Claude Codeのライフサイクル上の特定イベントで自動実行されるシェルコマンド（またはHTTP/MCP/プロンプト/エージェント）です。

SubAgentやSkillsが「LLMが必要と判断したら使うかもしれない」のに対し、Hooksはイベントが発火したら必ず実行されるのが本質的な違いです。これにより、lint・テスト・通知・監査ログのような「毎回確実に動かしたい処理」を、LLMの気まぐれに頼らず実現できます。

💡 設計判断のヒント 「Claudeに毎回指示しても忘れられる」処理はHooksに昇格させましょう。逆に「文脈次第で要否が変わる」処理はSkill/SubAgentが向いています。

PostCompact圧縮後の文脈再注入
⚠️ 仕様変更が起こりやすい領域 Hookイベントは現在も拡張が続いており（例: PostToolUseFailure, PermissionDenied, WorktreeCreateなどは比較的新しい追加）、最新の正確な一覧は必ず公式hooksリファレンスで確認してください。

3. 設定方法

3.1 配置場所

場所スコープ共有
~/.claude/settings.json全プロジェクト（自分）No
.claude/settings.jsonプロジェクトYes（Gitにコミット）
.claude/settings.local.jsonプロジェクトNo（gitignore）
Managed settings組織全体Yes（管理者制御）
Plugin hooks/hooks.jsonプラグインYes
Skill/Agent frontmatterコンポーネント実行中のみYes

3.2 設定スキーマ（3階層）

Hookの設定は イベント → matcher → handler の3階層構造です。

json{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

3.3 matcher（フィルタ）の仕様

  • "*" または省略 = 全マッチ
  • 英数字・_・|のみ = 完全一致または OR（例: Bash、Edit|Write）
  • その他の記号を含む = JavaScript正規表現として評価（例: mcp__memory__.*）

ツールイベントでは tool_name を、SubagentStart では agent_type を、というふうにイベントごとにmatchするフィールドが違います。

3.4 終了コードによる挙動制御

終了コード意味備考
0成功（判断なし）stdoutにJSONを出力すれば構造化制御可能
2ブロッキングエラーstderrがClaudeへのフィードバックになる。イベントによって挙動差あり
その他非ブロッキングエラー処理は続行するが警告がトランスクリプトに出る

3.5 JSON出力による高度な制御

終了コード0で stdoutにJSONを書くと、より細かい制御ができます。

json{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Use rg instead of grep for performance"
  }
}

PreToolUse の permissionDecision は "allow" / "deny" / "ask" の3値。UserPromptSubmitでは additionalContext でClaudeに追加情報を注入できます。

5. セキュリティ上の注意（必読）

🔒 セキュリティ警告 Hooksはあなたのシェル権限で任意のコマンドを実行できます。リポジトリをcloneしただけで .claude/settings.json 内のhookが動く可能性があるため、新しいリポジトリを開く前は必ず内容をレビューしてください。

5.1 信頼境界の管理

  • 未知のリポジトリを初めて開く前に cat .claude/settings.json でhook内容を確認
  • OSSのプルリクで .claude/ 配下の変更が来たときは特に慎重にレビュー
  • 外部プラグインを入れる前に hooks/hooks.json を確認
  • 組織配布時は Managed settings + allowManagedHooksOnly: true で個人hookをロックダウン

5.2 機密情報の扱い

  • APIキーやトークンはhookのcommand文字列に直書きしない。env変数経由で渡す
  • HTTP hookでは allowedEnvVars を明示し、漏洩リスクを限定
  • 監査ログを記録するhookはログのパーミッションに注意（chmod 600等）
  • UserPromptSubmitで秘密情報をstdoutに出すとClaudeのコンテキストに入り、Anthropicに送信されることを忘れない

5.3 権限とサンドボックス

  • hookスクリプトは適切に chmod +x し、所有者のみ書き込み可に（chmod 755程度）
  • ${CLAUDE_PROJECT_DIR} 等のplaceholderを使い、絶対パスでスクリプトを参照
  • 「破壊的操作（ファイル削除・git強制push等）」はhookには書かない。意図しないタイミングで動く
  • permission_modeが bypassPermissions でも、PreToolUse hookのdenyは効く（policy enforcementとして使える）

5.4 デッドロック・無限ループの回避

  • Stop hookで exit 2 を返し続けるとClaudeが止まれなくなる。8回連続ブロックで自動解除されるが、stop_hook_active フィールドを必ず確認
  • PostToolUse 内で同じツールを呼ぶ操作（書き換え→再書き換え）は無限ループの原因
⛔ やってはいけないこと 本ガイドでは破壊的なhookサンプル（rm -rf、強制push、外部送信スクリプト等）を意図的に掲載していません。これは「コピペして実害が出る」リスクを避けるためです。実運用ではドライランで検証し、まずローカルのsettings.local.jsonで試してから共有してください。

📝 この章の要点
Hooksは決定論的な拡張機構。イベント → matcher → handler の3階層で定義し、終了コード0/2/その他で挙動を制御する。PreToolUse は唯一ツール呼び出しをブロックできる。任意コマンドが実行できる強力な機構ゆえに、未知リポジトリのhookは必ずレビューすること。
✅ 理解度チェック（第3部）
  1. 「決定論的」とはHooksにおいて具体的に何を意味するか？
    解答を表示
    イベントが発火した瞬間に必ず該当hookが実行されること。LLMが「実行するかどうか」を判断する余地はない。lintや監査ログのように「絶対に漏らせない処理」に適する。
  2. PreToolUse hookが終了コード2で終了した場合、何が起きるか？
    解答を表示
    該当のツール呼び出しがブロックされ、stderrの内容がClaudeへのフィードバックとして返される。Claudeはそれを読んで別のアプローチに切り替えることができる。
  3. 新しいOSSをcloneして開く前にHooks観点で行うべきことは？
    解答を表示
    .claude/settings.json、.claude/hooks/、.claude/agents/などをまず目視レビューする。任意のシェルコマンドが含まれうるため、信頼できない場合は disableAllHooks: true を .claude/settings.local.json に設定してから開くのも有効。

実践演習

🎯 演習の進め方
  • 3つの演習はそれぞれ「課題 → ヒント → 解答例」の構成
  • 解答例は <details> で折りたたまれているので、まず自分で試してから開く

演習1: 自分用SubAgentを作る

課題: 「コミット前にgit diffを読み、コミットメッセージ案を3つ提案してくれる」SubAgentを作成し、.claude/agents/commit-suggester.md として配置せよ。

ヒント: 必要なツールは Bash（git diff用）と Read 程度。descriptionには「コミット前に提案を求められたとき」とトリガー条件を書く。

解答例を表示 
markdown---
name: commit-suggester
description: Suggests three commit message candidates based on the current staged diff. Use when the user is about to commit or asks for commit message ideas.
tools: Bash, Read
model: haiku
color: green
---

You are a commit message expert. When invoked:

1. Run `git diff --cached` to inspect the staged changes
2. If nothing is staged, run `git diff` to look at the working tree
3. Detect the project's commit convention (Conventional Commits, gitmoji, plain)
   by reading the last 20 commits with `git log --oneline -20`
4. Produce exactly 3 message candidates, ordered from most to least preferred
5. For each candidate, include: type (if conventional), short summary,
   and a one-line rationale

Do not commit yourself. Only suggest.

演習2: PostToolUseで自動lintを設定する

課題: Pythonファイル（*.py）をEditまたはWriteしたあとに、自動で ruff check --fix を実行するhookを .claude/settings.json に設定せよ。エラー時はClaudeへ通知すること。

ヒント: matcherは Edit|Write。if フィールドで拡張子を絞れる。lint失敗時は exit 0（PostToolUseはブロック不可だがstderrを通じてClaudeに伝わる）。

解答例を表示 
json{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "if": "Edit(*.py)",
            "command": "FILE=$(jq -r '.tool_input.file_path'); ruff check --fix \"$FILE\" 1>&2 || echo \"ruff reported issues in $FILE\" 1>&2"
          },
          {
            "type": "command",
            "if": "Write(*.py)",
            "command": "FILE=$(jq -r '.tool_input.file_path'); ruff check --fix \"$FILE\" 1>&2 || echo \"ruff reported issues in $FILE\" 1>&2"
          }
        ]
      }
    ]
  }
}

※ if は1ルールずつしか書けないため、Edit用とWrite用で2エントリに分けています。

演習3: SubAgent + Hooksの組み合わせワークフロー

課題: 以下を満たす小さなワークフローを構築せよ。

  1. 「security-scanner」というSubAgentを作る（依存関係チェックとシークレットスキャン用、読み取り専用ツールのみ）
  2. SubAgentが終了したら（SubagentStop）、結果を ~/.claude/audit/scan-YYYYMMDD.log に追記する
  3. matcherでこのSubAgent以外には反応しないようにする
解答例を表示

① .claude/agents/security-scanner.md

markdown---
name: security-scanner
description: Scans the project for known-vulnerable dependencies and accidental secret commits. Read-only. Use proactively before opening a PR.
tools: Read, Grep, Glob, Bash
model: sonnet
color: purple
---

You are a security scanner. Perform these checks and report findings:

1. Look for committed secrets using grep patterns like
   `AWS_SECRET`, `PRIVATE KEY`, `BEGIN RSA`, `api[_-]?key\s*=`
2. Inspect dependency manifests (package.json, requirements.txt, Cargo.toml)
   and flag versions known to be vulnerable in well-known advisories
3. Look for sensitive files (`.env*`, `id_rsa`, `*.pem`) tracked in git
   using `git ls-files`

Output a structured summary:
- CRITICAL: must fix before merge
- WARNING: should investigate
- INFO: informational

Do not modify files. Read-only.

② .claude/settings.json

json{
  "hooks": {
    "SubagentStop": [
      {
        "matcher": "security-scanner",
        "hooks": [
          {
            "type": "command",
            "command": "mkdir -p ~/.claude/audit && jq -c '{ts: now|todate, agent: .agent_type, session: .session_id}' >> ~/.claude/audit/scan-$(date +%Y%m%d).log"
          }
        ]
      }
    ]
  }
}

この組み合わせにより、scanner起動 → 結果報告 → ログ記録の流れが自動化されます。matcherを "security-scanner" に絞っているので、他のSubAgent終了時には発火しません。

Claude Codeのライフサイクル上の特定イベントで自動実行されるシェルコマンド等。決定論的にlint・通知・ブロックなどを行える。
Skills
指示書・手順書をMarkdownでパッケージ化し、Claudeが必要に応じて読み込む機構。Agent Skillsオープン標準に準拠。SubAgentと違いメインコンテキスト内で動作。
Slash Commands
/で始まるユーザー入力コマンド。/agents /hooks /configなど。
MCP（Model Context Protocol）
外部システム（DB、Slack、Playwrightなど）をClaudeのツールとして接続するためのオープンプロトコル。
CLAUDE.md
Claudeに永続的な指示・コンテキストを与えるMarkdownファイル。プロジェクト・ユーザー・ローカル階層がある。
Permission Mode
権限プロンプトの挙動を制御するモード。default / acceptEdits / auto / dontAsk / bypassPermissions / plan。
Matcher
Hookの発火条件を絞るフィルタ。ツール名・正規表現・OR記法（|）が使える。
YAML Frontmatter
Markdownファイル冒頭の ---〜--- で囲まれたメタデータブロック。SubAgentやSkillの設定を記述する。
コンテキスト（Context Window）
LLMが一度に処理できる入力・出力トークンの上限。大量のログを直接読むと圧迫されるため、SubAgentに分離するのが定石。
ヘッドレスモード
claude -p "..." で起動する非対話モード。CI/CDやスクリプトから呼ぶ用途。
Fork（フォーク）
SubAgentの一種で、親の会話履歴をすべて引き継いで起動するモード。/forkで起動。
Managed Settings
組織管理者が配布する、ユーザーが上書きできない最高優先度の設定。
