CLAUDE.md と Agent Skills 学習ガイド

Claude Code の二大カスタマイズ機構である「CLAUDE.md（メモリ）」と「Agent Skills」を、概念理解から実装・チーム運用までを一気通貫で学べる教材です。

対象読者: Claude Code を業務利用するエンジニア / DevOps / チームリード 所要時間: 約 90〜120 分（演習込み） 最終更新: 2026-05-22 前提: Markdown / Git の基本知識

このガイドの読み方

学習目標

  • CLAUDE.md と Skills の役割の違いを一言で説明できる。
  • 自分のプロジェクトに最小構成の CLAUDE.md と Skill を作成できる。
  • 状況に応じて、CLAUDE.md / Skill / サブエージェント / スラッシュコマンドのどれを使うべきか判断できる。
  • チームでの安全な共有・運用のポイントを把握する。

推奨学習ルート

情報の出典について

このガイドでは記述ごとに出典を明示します。公式 は Anthropic / Claude 公式ドキュメント由来、参考 はコミュニティ・記事由来、補足 は環境差異や注意点です。

第1部 — CLAUDE.md

1.1 CLAUDE.md とは何か 公式

この節の学習目標

  • CLAUDE.md の役割と「いつ読み込まれるか」を説明できる。
  • auto memory との違いを把握する。

CLAUDE.md とは、Claude Code がセッション開始時に自動で読み込む Markdown ファイルで、あなたが Claude に持たせたい「永続的な指示・コンテキスト」を記述する場所です。書き方は普通の Markdown で、Claude にとっての長期記憶として機能します。

Claude Code は毎回新しいコンテキストで起動するため、プロジェクトのビルドコマンドや命名規則、アーキテクチャ上の決定事項などを、CLAUDE.md を通じて毎回伝える必要があります。これを書いておくと、毎セッション同じ説明をする手間が省け、Claude の出力品質が安定します。

💡 ヒント — auto memory との違い

公式ドキュメントによれば、Claude Code には CLAUDE.md と並行して auto memory という仕組みもあります。CLAUDE.md はあなたが書く指示、auto memory はClaude が自動で書き溜める学習メモです。auto memory は ~/.claude/projects/<project>/memory/MEMORY.md に保存され、起動時に先頭 200 行（または 25KB）まで読み込まれます。

なぜ必要か

CLAUDE.md は次のような場面で価値を発揮します。

  • Claude が同じミスを二回した。
  • コードレビューで Claude が知っておくべきだった指摘を受けた。
  • 毎セッション同じ「補正・前提」をチャットに打ち込んでいる。
  • 新しいチームメンバーに同じコンテキスト共有が必要。

読み込まれるタイミング

CLAUDE.md はセッション開始時に自動的にコンテキストへ注入されます。サブディレクトリの CLAUDE.md は、Claude がそのディレクトリ配下のファイルを読んだときにオンデマンドで読み込まれます。

⚠️ 注意

CLAUDE.md の内容はシステムプロンプトの一部ではなく、ユーザーメッセージとして配信されます。Claude は読んで従おうとしますが、強制ではありません。曖昧な指示や矛盾する指示には従わないことがあります。「必ず実行されるべき手順」は hooks として書きましょう。

📝 理解度チェック 1.1

Q1. CLAUDE.md と auto memory の最大の違いは？

A. 書き手が異なります。 CLAUDE.md は人間がプロジェクト規約や方針を書き込み、auto memory は Claude 自身が会話中の修正や好みを観察して書き溜めます。

Q2. CLAUDE.md はいつ読み込まれる？

A. セッション開始時に自動で読み込まれ、コンテキストに注入されます。サブディレクトリの CLAUDE.md は、Claude がそのディレクトリ配下のファイルを読んだときにオンデマンド読み込みされます。

Q3. CLAUDE.md は「絶対的なルール」として扱われる？

A. いいえ。 設定ではなくコンテキストとして扱われるため、Claude が必ず従う保証はありません。確実な強制が必要なら hooks や permissions.deny を使います。

1.2 配置場所と階層 公式

この節の学習目標

  • 4 つの主な配置場所とスコープを把握する。
  • @ インポートと CLAUDE.local.md の使い方を理解する。

4 つのスコープ

CLAUDE.md は複数の場所に置けます。下にあるものほどスコープが具体的で、プロジェクト固有の指示はユーザーグローバル指示の後に読まれます（=後勝ち的に効きやすい）。

スコープ配置場所用途共有範囲
Managed policy OS固有のシステムパス
(例: macOS は /Library/Application Support/ClaudeCode/CLAUDE.md)		 組織全体の指示（IT/DevOps 管理） マシン上の全ユーザー
User ~/.claude/CLAUDE.md 個人の好み・全プロジェクト共通 自分のみ・全プロジェクト
Project ./CLAUDE.md または ./.claude/CLAUDE.md プロジェクト規約・アーキテクチャ チーム（Git 経由）
Local ./CLAUDE.local.md 個人のプロジェクト固有設定 自分のみ・該当プロジェクト

読み込み順

Claude Code はカレントディレクトリからファイルシステムのルートに向かって歩き、各ディレクトリで CLAUDE.md と CLAUDE.local.md を探します。見つかったすべてが連結されてコンテキストに入ります。順序は「ルートに近い方が先、現在地に近い方が後」、同じディレクトリ内では CLAUDE.local.md が CLAUDE.md の後に追加されます。

CLAUDE.local.md ― Git に含めない個人設定

個人専用のサンドボックス URL、好みのテストデータ、ローカル DB の接続文字列などは CLAUDE.local.md に書き、.gitignore に追加します。/init を実行して「personal」を選ぶと自動で .gitignore 追記されます。

@インポートで他ファイルを取り込む

CLAUDE.md は @path/to/file 構文で他のファイルをインポートできます。相対パスは「そのファイルからの相対」で、絶対パスや ~/ も使えます。インポートされたファイルもセッション開始時にコンテキストへ展開されます（最大 5 階層）。

# Project Conventions

See @README.md for project overview and @package.json for npm scripts.

## Git workflow
- See @docs/git-instructions.md

## My personal preferences (worktree-wide)
- @~/.claude/my-project-instructions.md

AGENTS.md との互換

他のコーディングエージェント用に AGENTS.md を既に使っている場合は、CLAUDE.md 側で @AGENTS.md をインポートすれば両方のツールで同じ指示を共有できます。Claude 固有の追記はその下に書けます。

@AGENTS.md

## Claude Code

Use plan mode for changes under `src/billing/`.
⚠️ 落とし穴 — モノレポでの読み込み

大規模モノレポでは、関係ない他チームの CLAUDE.md が祖先ディレクトリから読み込まれてノイズになることがあります。.claude/settings.local.json に claudeMdExcludes を設定して除外しましょう。

📝 理解度チェック 1.2

Q1. プロジェクトの CLAUDE.md とユーザーの CLAUDE.md、両方ある時はどちらが優先？

A. 両方読まれます。 上書きではなく連結です。読まれる順序はユーザー → プロジェクトの順で、後に読まれたプロジェクト指示の方が直近に効きやすくなります。

Q2. 個人の API キーや一時 URL を CLAUDE.md に書いて良い？

A. いけません。 CLAUDE.md は Git にコミットされる前提です。個人情報や秘密情報は CLAUDE.local.md（gitignore 済み）に書きます。

Q3. README.md の内容を CLAUDE.md でも参照させたい。重複コピーすべき？

A. 不要です。 @README.md でインポートできます。最大 5 階層までネスト可能です。

1.3 書き方の基本 公式

この節の学習目標

  • 効果的な CLAUDE.md の構成要素を把握する。
  • 「具体的・簡潔・一貫」の3原則を理解する。

基本構文

CLAUDE.md は普通の Markdown です。見出しと箇条書きでセクションを整理しましょう。公式ガイダンスによれば 200 行以内を目標とし、それを超えると遵守率が下がると説明されています。

推奨セクション構成

典型的な CLAUDE.md には次のような項目が含まれます。参考

  • プロジェクト概要: 何を作っているか、技術スタック
  • ビルド & テストコマンド: npm test, make lint など
  • コーディング規約: インデント、命名、Lint ルール
  • アーキテクチャ: ディレクトリ構造、依存関係、設計判断
  • 避けるべきパターン: deprecated な API、過去の失敗事例
  • 共通ワークフロー: PR の出し方、ブランチ命名

3つの原則

原則悪い例良い例
具体性「コードを綺麗に書く」「2スペースインデント。1行 100 文字以内」
検証可能「テストを書く」「commit 前に npm test を実行」
場所の明示「ファイルを整理」「API ハンドラは src/api/handlers/ に置く」

HTML コメントで「人間用メモ」を残す

ブロックレベルの HTML コメント（<!-- ... -->）はClaude のコンテキストに注入される前に除去されます。メンテナー向けメモを書いてもトークンを消費しません（コードブロック内のコメントは保持されます）。

💡 対話中に育てるテクニック

会話の途中で # プレフィックスを使うと、Claude にメモを CLAUDE.md に追記させられます（"Add this to CLAUDE.md" と頼んでも同じ）。/memory コマンドで現在読み込まれているメモリファイルを確認・編集できます。

.claude/rules/ によるモジュール化

大規模プロジェクトでは指示を1ファイルにまとめず、.claude/rules/ ディレクトリに分割できます。さらに各ファイルの YAML frontmatter で paths を指定すると、該当ファイルを編集している時だけ読み込まれる「パス特化型ルール」になります。

---
paths:
  - "src/api/**/*.ts"
---

# API Development Rules

- All API endpoints must include input validation
- Use the standard error response format
- Include OpenAPI documentation comments

📝 理解度チェック 1.3

Q1. CLAUDE.md は何行くらいまでに収めるべき？

A. 公式は200 行以内を推奨。長くなるとトークン消費と遵守率の両面で不利になります。

Q2. API ハンドラ用の規約だけ、API ファイル編集時に限定して読み込ませたい。どうする？

A. .claude/rules/api.md を作り、frontmatter に paths: ["src/api/**/*.ts"] を書く。

Q3. CLAUDE.md にメンテナー向けのメモを書きたい。どうすればトークンを使わずに残せる？

A. ブロックレベルの HTML コメント <!-- メモ --> を使う。Claude のコンテキストに入る前に除去されます。

1.4 ベストプラクティス 公式 参考

この節の学習目標

  • DO / DON'T を明確に区別できる。
  • /init での雛形生成と「育てる」プロセスを把握する。

DO（やるべきこと）

  • 具体的・検証可能な指示を書く（"npm run build を git commit 前に実行"）。
  • プロジェクト固有のコマンド・命名規則・避けるべきパターンを明文化。
  • 「同じ修正・補正を2回以上した」時点で CLAUDE.md に書き起こす。
  • 定期的に見直し、矛盾・古い記述を削除する。

DON'T（避けるべきこと）

  • 冗長な一般論（"良いコードを書く" など）を書く。
  • 頻繁に変わる情報（バージョン番号、現在のスプリント目標）を直接書く。
  • 機密情報や個人 URL を含める。→ CLAUDE.local.md へ
  • 1ファイルに 200 行を大きく超える内容を詰め込む。
  • 多段階の手順を CLAUDE.md に書く。→ Skill に切り出す

/init で雛形を作って育てる

新規プロジェクトでは Claude Code 内で /init を実行すると、リポジトリを解析して CLAUDE.md の雛形を自動生成してくれます。AGENTS.md や .cursorrules, .windsurfrules があれば、それらの内容も読んで取り込みます。公式

雛形生成後は、作業中に気づいた点を # プレフィックスや「これを CLAUDE.md に追記して」という依頼で少しずつ育てるのが定着パターンです。

⚠️ /compact 後の挙動

会話が長くなって /compact でコンテキストを圧縮すると、プロジェクトルートの CLAUDE.md は再注入されますが、サブディレクトリのネスト CLAUDE.md は再注入されません（次回そのディレクトリのファイルを読んだ時に再ロード）。会話中だけで与えた指示は失われるので、重要なら CLAUDE.md に書き戻しましょう。

📝 理解度チェック 1.4

Q1. /init は何をする？

A. リポジトリ構造を解析し、CLAUDE.md の雛形を生成します。既存の AGENTS.md / .cursorrules / .windsurfrules があれば、その内容も取り込みます。

Q2. 多段階のデプロイ手順を CLAUDE.md に詰め込んでしまった。どうする？

A. Skill（特に disable-model-invocation: true 付き）として切り出す。CLAUDE.md は「ファクト」に絞り、「プロシージャ」は Skill に置くのが公式の推奨です。

第2部 — Agent Skills

2.1 Skills とは何か 公式

この節の学習目標

  • Skill の正体（命令・スクリプト・リソースのパッケージ）を理解する。
  • CLAUDE.md / サブエージェント / スラッシュコマンドとの違いを言える。
  • 「progressive disclosure」のメンタルモデルを掴む。

Agent Skills は、特定タスクのための「指示・メタデータ・任意のスクリプトやテンプレート」を 1 つのフォルダにまとめた拡張機能です。Claude は必要なときだけ自動で読み込んで使用します。Anthropic は "Skills は新入社員のオンボーディングガイドのようなもの" と表現しています。

2025 年 12 月 18 日に agentskills.io としてオープン標準化されました。複数の AI ツールで利用可能な仕様です。公式

CLAUDE.md との根本的な違い

CLAUDE.mdSkill
読み込み常時（毎セッション全文）オンデマンド（必要時のみ本文を読む）
トークン消費毎回課金されるメタデータのみ常時、本文はトリガー時
適した用途ファクト・規約（常識として持たせたい情報）多段階手順・専門タスク（必要時に呼び出す能力）
サイズ目安200 行以内SKILL.md 本文 500 行以内、参照ファイルは事実上無制限
スクリプト同梱×○（Python/Shell など）
📐 メンタルモデル

CLAUDE.md＝Claude が常に身につけている「教科書」。
Skill＝図書館にある「専門書」。必要な時だけ取り出して開く。
サブエージェント＝別の部屋にいる「専門スタッフ」。仕事を頼んで結果だけもらう。
スラッシュコマンド（builtin）＝固定機能のボタン。

関連機能との違い（早見表）

機能本質呼び出し方独立コンテキスト
CLAUDE.md常時コンテキスト自動（起動時）×（メイン）
Skillオンデマンド能力自動 or /skill-name×（通常はメイン、context: fork で○）
サブエージェント独立コンテキストの作業者Claude が委譲 / @agent-... / --agent○
スラッシュコマンド（builtin）固定機能/command機能による

📝 理解度チェック 2.1

Q1. 「リリースノートを書く手順」は CLAUDE.md と Skill のどちらに書くべき？

A. Skill。多段階の手続きであり、毎セッション必要ではないからです。CLAUDE.md には書かないのが公式の指針。

Q2. progressive disclosure とは何を指す？

A. 必要な情報を必要な時にだけ読み込む方式。Skill は L1: name+description（常時） → L2: SKILL.md 本文（トリガー時） → L3: 参照ファイル/スクリプト（更に必要な時のみ） の三段階。

Q3. ある手順を Skill にすべきか、サブエージェントにすべきか迷っている。判断軸は？

A. 独立コンテキストが欲しいかどうか。大量のログや検索結果を取り扱い、メインの会話に汚染を持ち込みたくないならサブエージェント。メインの会話の中で再利用可能な手順として動かしたいなら Skill。

2.2 構造とファイル形式 公式

この節の学習目標

  • SKILL.md のフロントマターと必須項目を把握する。
  • 補助ファイル（スクリプト、リファレンス、テンプレート）の同梱方法を理解する。

ディレクトリ構成

Skill は 1 フォルダ ＝ 1 Skill です。最低限 SKILL.md が必要で、補助ファイルは自由に追加できます。

my-skill/
├── SKILL.md           # 必須: メイン指示 + frontmatter
├── reference.md       # 任意: 詳細リファレンス（必要時のみ読まれる）
├── examples.md        # 任意: 出力例
└── scripts/
    └── helper.py      # 任意: 実行スクリプト（中身はcontextに入らない）

SKILL.md のフロントマター

YAML frontmatter（--- で囲まれた部分）でメタデータを宣言します。Claude API / claude.ai 側の標準仕様では name と description が必須で、Claude Code ではどちらも省略可能（ただし description は強く推奨）です。

---
name: processing-pdfs
description: Extracts text and tables from PDFs, fills forms, and merges documents. Use when the user mentions PDF files or attaches a .pdf.
---

# Processing PDFs

When the user provides a PDF file:
1. Read it with `pdftotext` if available, or `pdfplumber` otherwise.
2. ...

主要フロントマターフィールド

フィールド必須説明
nameAPI/claude.ai: 必須
Claude Code: 任意		小文字英数字とハイフンのみ。最大 64 文字。"anthropic" / "claude" は予約語で使用不可。Claude Code ではディレクトリ名がデフォルト。
description推奨Skill が何をするか + いつ使うか。最大 1024 文字（Claude Code の listing では when_to_use と合わせて 1536 文字でトランケート）。これが選択精度を左右する。
when_to_use任意Claude Code 拡張。トリガーフレーズなど。description の補足。
disable-model-invocation任意true にすると Claude による自動呼び出しを禁止し、/name による手動のみに。デプロイ系で推奨。
user-invocable任意false にすると / メニューから隠れる（背景知識用）。
allowed-tools任意この Skill が active な間、許可なしで使えるツール。例: Bash(git add *) Bash(git commit *)
context任意fork を指定するとサブエージェントの独立コンテキストで実行。
agent任意context: fork 時に使うエージェント種別（Explore / Plan / general-purpose など）。
paths任意glob パターン。マッチするファイルを Claude が読んだ時にのみ自動有効化。
arguments / argument-hint任意Claude Code 拡張。位置引数の名前定義と autocomplete ヒント。
model / effort任意Claude Code 拡張。Skill 実行中のモデル/努力レベルの上書き。
hooks任意Claude Code 拡張。この Skill のライフサイクルに紐づくフック。
⚠️ 落とし穴 — description が選択精度を決める

Claude は description を読んで「いまの依頼でこの Skill を使うべきか」を判断します。曖昧な記述（"Helper for files"）だと発火しません。「何をするか」と「いつ使うか」を必ず三人称で書きましょう（公式: "Processes Excel files and generates reports" ◯ / "I can help you process Excel files" ✕）。

補助ファイルの参照

大きなリファレンスは別ファイルに分けて、SKILL.md 内から参照します。リンクが1 階層になるよう保つのがコツです（深くネストするとプレビュー読みで欠落しやすい）。

## Additional resources

- For complete API details, see [reference.md](reference.md)
- For form-filling guidance, see [forms.md](forms.md)

## Validation script

📝 理解度チェック 2.2

Q1. SKILL.md のフロントマターで最重要なフィールドは？

A. description。Claude が「この Skill を使うべきか」を判断するための唯一の根拠です（API/claude.ai では name も必須）。

Q2. 200 ページのリファレンスを Skill に同梱したい。コンテキストを圧迫しない？

A. しません。 補助ファイルは「ファイルシステム上に置かれ、Claude が必要な時に bash で読む」だけ。読まなければ 0 トークン。これが progressive disclosure の利点です。

2.3 動作の仕組み 公式

この節の学習目標

  • 3段階の Progressive Disclosure を図で理解する。
  • Skill が発火するタイミングと、コンテキストへの影響を把握する。

3段階のロード

レベル読み込みタイミングトークンコスト内容
L1 メタデータ常時（起動時）～100 トークン/Skillfrontmatter の name + description
L2 指示本文Skill 発火時～5K トークンSKILL.md の本文
L3 リソース/スクリプト必要時のみ事実上 0 〜 任意参照ファイル、スクリプト実行結果

動作フロー（PDF Skill の例）

Skill content の寿命

Claude Code では、Skill が一度呼ばれるとその内容は単一のメッセージとして会話に挿入され、セッション中ずっと残ります（再読込はされない）。/compact 時には、各 Skill の最新呼び出し分から先頭 5,000 トークンが再アタッチされ、合計 25,000 トークンの予算が割り当てられます。

⚠️ Skill が「効かなくなった」と感じたら

大抵は内容がまだコンテキストにあるが、モデルが他のツール/アプローチを選んでいるだけです。対策は (1) description を強化、(2) Skill を再呼び出ししてフレッシュに、(3) 確実性が必要なら hooks で決定論的に強制。

📝 理解度チェック 2.3

Q1. Skill を 100 個インストールしたら起動が遅くなる？

A. 起動時にロードされるのは各 Skill の name+description（約100トークン）だけなので、影響は小さいです。ただし description が長いと skillListingBudgetFraction（context の 1%）を超えてトランケートされることがあり、選択精度に影響します。

Q2. Skill のスクリプトファイルの中身はコンテキストに入る？

A. 入りません。 Claude はスクリプトを bash で実行するだけで、出力のみがコンテキストに入ります。これがスクリプト同梱の大きな利点です。

2.4 作り方（ハンズオン） 公式

この節の学習目標

  • 最小構成の Skill を 5 分で作成できる。
  • 「Conventional Commits 風の git commit 作成」 Skill を完成させる。

ステップ1: ディレクトリと SKILL.md を作る

プロジェクトローカルで作るなら .claude/skills/<name>/、自分用なら ~/.claude/skills/<name>/ です。今回はプロジェクトに置きます。

# Bash
mkdir -p .claude/skills/conventional-commit
$EDITOR .claude/skills/conventional-commit/SKILL.md

ステップ2: SKILL.md を書く

---
name: conventional-commit
description: Creates a Conventional Commits style git commit message from the currently staged changes. Use when the user asks to commit, stage and commit, or write a commit message.
disable-model-invocation: true
allowed-tools: Bash(git status:*) Bash(git diff:*) Bash(git add:*) Bash(git commit:*)
---

# Conventional Commit

Create a Conventional Commits style commit for the currently staged changes.

## Steps

1. Run `git status --short` to see what is staged.
2. Run `git diff --cached` to inspect the actual diff.
3. Decide the type from the diff:
   - `feat`     — new user-facing feature
   - `fix`      — bug fix
   - `refactor` — internal restructure, no behavior change
   - `docs`     — documentation only
   - `test`     — test changes only
   - `chore`    — build/config/deps
4. Pick a scope from the top-level changed directory (e.g. `api`, `web`).
5. Write a subject in imperative mood, <= 72 chars.
6. Optionally add a body explaining "why".
7. Show the proposed message to the user. Wait for approval.
8. After approval, run `git commit -m "<subject>" -m "<body>"`.

## Examples

- `feat(api): add cursor pagination to /users`
- `fix(web): prevent double-submit on login form`
- `refactor(billing): extract pricing calculator`
- `docs: explain CLAUDE.md import syntax in README`

ステップ3: 動作確認

  1. Claude Code を再起動（または .claude/skills/ を新規作成した場合のみ必要）。
  2. /skills コマンドで Skill が一覧に出るか確認。
  3. 何かファイルを編集し git add。
  4. Claude Code 内で /conventional-commit を実行。
💡 ヒント — disable-model-invocation の使いどき

commit / deploy / send-notification など副作用のある操作は disable-model-invocation: true にすべきです。Claude がコードを見て「準備完了っぽい」と判断して勝手にデプロイしないように。

ステップ4: スクリプトを同梱して確実性を上げる

たとえば「diff から自動的に scope を推定するスクリプト」を同梱して、Claude にそれを呼ばせれば、毎回同じロジックで決まります。

# .claude/skills/conventional-commit/scripts/guess-scope.sh
#!/usr/bin/env bash
set -euo pipefail

git diff --cached --name-only \
  | awk -F/ '{ print $1 }' \
  | sort -u \
  | head -n 1

SKILL.md 側で bash ${CLAUDE_SKILL_DIR}/scripts/guess-scope.sh を呼ぶよう指示すれば、Claude が同じスクリプトを実行するため挙動が安定します。スクリプトのコード自体はコンテキストに入りません。

📝 理解度チェック 2.4

Q1. プロジェクトメンバー全員で Skill を共有したい。どこに置く？

A. .claude/skills/<name>/ に置き、Git にコミットします。ユーザーグローバル（~/.claude/skills/）は自分専用です。

Q2. 新しく作った Skill が / メニューに出てこない。考えられる原因は？

A. (1) user-invocable: false を付けている、(2) .claude/skills/ がセッション開始時に存在しなかったので watch されていない（Claude Code 再起動が必要）、(3) frontmatter の YAML が壊れている、(4) skillOverrides で "off" に設定されている、など。

2.5 ベストプラクティス 公式

この節の学習目標

  • 良い description の書き方を身につける。
  • 1 Skill 1 責務、適切な「自由度」、評価駆動開発の考え方を理解する。

description は「三人称・具体・トリガー語句入り」

❌ 避けるべき
description: I can help with PDFs.

description: Helper for working with files.

description: You can use this for spreadsheets.
✅ 推奨
description: Extracts text and tables from PDFs, fills forms, and merges documents. Use when the user mentions PDF files or attaches a .pdf.

description: Analyzes Excel files (.xlsx, .xls), generates summary reports, and creates charts. Use when the user asks about spreadsheets or attaches an Excel file.

命名規約 — gerund form 推奨

公式は動詞 + -ing の動名詞形を推奨しています:

  • ✅ processing-pdfs / analyzing-spreadsheets / managing-databases / writing-documentation
  • △ 代替可: pdf-processing, spreadsheet-analysis
  • ❌ 避ける: helper, utils, tools, data

1 Skill 1 責務

「あらゆるドキュメントを扱う」Skill より、「PDF を扱う」「Word を扱う」「Excel を扱う」と分けたほうが、選択精度も保守性も上がります。

「自由度」を場面に合わせる

自由度使うとき書き方
高複数アプローチが有効/コンテキスト依存テキストベースの方針・ヒューリスティック
中推奨パターンはあるが変動可疑似コード or パラメータ付きスクリプト
低壊れやすい/厳密な順序が必要固定の手順 + 検証スクリプト

評価駆動開発（Evaluation-driven）

公式の推奨フローは:

  1. ギャップ特定: Skill なしで Claude にタスクを与え、失敗する箇所を観察。
  2. 評価作成: 失敗パターンを 3 シナリオで再現できる評価を作る。
  3. ベースライン測定: Skill なしのスコアを記録。
  4. 最小指示を書く: 失敗を埋めるのに必要十分な内容で SKILL.md を作る。
  5. 反復: 評価を回しスコアを比較、SKILL.md を改善。

Claude と協力して Skill を作る

公式は "Claude A（設計役）と Claude B（実行役）を使い分ける" パターンを推奨しています。あるタスクを Claude A と一緒に終えたあと、「いまの手順を Skill にまとめて」と頼みます。次に Claude B（新しいセッション）で動作確認し、ズレを Claude A にフィードバックして調整します。

その他のチェックポイント

  • SKILL.md 本文は500 行以内を目安。それを超える内容は別ファイルに分割。
  • 用語は一貫させる（"field" と "box" を混在させない）。
  • 時限情報（"2024 年現在の API バージョン" など）を入れない。または "old patterns" 節に隔離。
  • パスは forward slash（scripts/x.py）。Windows でも動くように。
  • 参照ファイルは1 階層まで（深いネスト避ける）。
  • 長いリファレンスファイル（100 行超）には目次をつける。
  • MCP ツールは ServerName:tool_name のフルネームで参照。
  • Haiku / Sonnet / Opus 全部でテストする（モデルにより必要な詳細度が変わる）。
⚠️ セキュリティ上の重大な落とし穴

Skill は「実質的にコード入りパッケージ」です。信頼できない発信元の Skill をインストールしてはいけません。allowed-tools で広範な権限を自分自身に与えうるため、プロジェクトの .claude/skills/ をワークスペースとして信頼する前に必ず中身をレビューしましょう。

./.claude/settings.json✅ するプロジェクト共通設定（permissions など）
./.claude/settings.local.json❌ しない個人ローカル設定
~/.claude/**—そもそも個人ホームディレクトリ

典型的な .gitignore

# Claude Code - personal overrides
CLAUDE.local.md
.claude/settings.local.json

# 任意: 個人用ローカル Skill を Git に入れたくない場合
# .claude/skills/personal-*/
⚠️ workspace trust に注意

プロジェクトの .claude/skills/ に allowed-tools 付きの Skill が含まれていると、Claude Code はそのワークスペースを「信頼するか？」確認します。Skill は実質的に実行コードであることを忘れずに、レビューしてから信頼を与えましょう。

4.2 レビュー・更新サイクル

育てるためのトリガー

次のような時に CLAUDE.md / Skill を更新する習慣をつけます:

  • コードレビューで「Claude にも知っておいてほしかった」指摘を受けた → CLAUDE.md に追記。
  • 同じ手順を 2 回以上手で説明した → Skill に切り出す。
  • Skill が期待通り発火しない → description を強化、または評価を追加。
  • 四半期に1回、CLAUDE.md と Skill 一覧を読み直して、古い・矛盾する記述を整理。

PR ベースのレビュー

  • CLAUDE.md / Skill の変更は通常のコードと同じく PR レビュー対象に。
  • レビュー観点: 具体的か、検証可能か、200/500 行に収まっているか、機密情報が含まれていないか、allowed-tools が広すぎないか。
  • Skill 追加時は動作確認のスクショや実行ログを PR に添えると良い。

計測とデバッグ

  • /memory — 現在ロードされている CLAUDE.md / rules を確認。
  • /skills — 利用可能な Skill を一覧（tでトークン量ソート、Spaceで可視性切替）。
  • /doctor — description のバジェット溢れなど設定の健全性を診断。
  • /context — 現在のコンテキスト使用状況を可視化。

4.3 チーム内で標準化するコツ

  • READMEで存在を周知: 新規メンバーが ./.claude/ を見つけられるよう、READMEに「このプロジェクトは Claude Code を活用しており、CLAUDE.md と .claude/skills/ を参照しています」と書く。
  • 命名規約を統一: Skill 名は gerund 形式（processing-pdfs 等）に統一する旨を CLAUDE.md or 内部 wiki に明記。
  • テンプレートを用意: .claude/skills/_template/ に SKILL.md のひな形を置いておく。
  • /team-onboarding を活用: Claude Code 内蔵コマンドで、過去 30 日の使用履歴からチームオンボーディングガイドを自動生成できる。公式
  • プラグイン化の検討: 複数プロジェクトで共有したい Skill 群はプラグインとして配布。anthropics/skills のように /plugin marketplace add で配布可能。

4.4 セキュリティ 公式

この節の学習目標

  • CLAUDE.md / Skill に書いてはいけない情報を理解する。
  • 外部由来 Skill の取り扱い方針を持つ。
⛔ 書いてはいけないもの
  • API キー、トークン、パスワード、接続文字列の中の認証情報
  • 本番 URL（社外秘な場合）
  • 個人を特定できる情報（顧客名、メール）
  • 未公開機能・契約上の機密

必要な場合は CLAUDE.local.md や OS の secret manager / 環境変数を経由し、CLAUDE.md には環境変数名だけ記載するパターンが安全です。

外部 Skill 受け入れの方針

公式は「信頼できる発信元の Skill のみインストール」を明確に求めています。受け入れ前のチェック観点:

  • SKILL.md / 補助ファイル / スクリプトをすべて読む（特にネットワーク呼び出し、ファイル削除、外部 API へのデータ送信）。
  • allowed-tools の範囲を確認。Bash など広い権限を与えていないか。
  • 外部 URL から動的に内容を取得する Skill は要注意（取得先が乗っ取られると Skill が事実上書き換わる）。
  • 本番系・機密データへの接続権限がある環境では特に慎重に。

組織レベルの統制

  • Managed settings で claudeMd や permissions.deny を中央配布。
  • disableSkillShellExecution: true で Skill 内の !`cmd` 実行を一括禁止。
  • permissions.deny に Skill(deploy *) など特定 Skill のブロックを設定。

📝 理解度チェック 第4部

Q1. CLAUDE.local.md をうっかり Git にコミットしてしまった。何をすべき？

A. 含まれる機密情報（API キー等）があれば即座にローテーション。git rm --cached で履歴から除外（必要に応じて git filter-repo 等で過去履歴も洗浄）、.gitignore に追加。

Q2. 外部の人が作った Skill を入れる前にチェックすべき3点は？

A. (1) SKILL.md と全スクリプトを目視レビュー、(2) allowed-tools が広すぎないか、(3) 外部 URL からの動的取得（特に !`curl ...` など）がないか。

🎯 実践演習

ここまでで学んだ内容を、実際に手を動かして定着させましょう。各演習には参考解答例がついています。

演習1（初級）— あなたのプロジェクトに最小の CLAUDE.md を作る

所要時間: 15 分 / 到達点: /init で雛形を作り、最低 5 セクションを埋める。

  1. 適当なプロジェクトで Claude Code を起動し、/init を実行。
  2. 生成された CLAUDE.md に次のセクションが含まれているか確認:
    • プロジェクト概要 / 技術スタック
    • ビルド・テスト・Lint のコマンド
    • コーディング規約（インデント・命名）
    • ディレクトリ構成の重要点
    • 避けるべきパターン（DON'T）
  3. 抜けがあれば手書きで埋め、200 行を超えないようにする。
📖 参考解答（Node.js + Express の例） 
# MyAPI

Node.js 20 + Express + Postgres (via Knex). REST API for internal tools.

## Commands
- `pnpm dev`        - run with hot reload
- `pnpm test`       - vitest, must pass before commit
- `pnpm lint`       - eslint + prettier check
- `pnpm migrate`    - run knex migrations

## Conventions
- 2-space indent, single quotes, trailing comma
- Files: kebab-case. Functions/vars: camelCase
- One route handler per file under `src/routes/`
- Always validate request body with Zod

## Architecture
- `src/routes/`      Express routers
- `src/services/`    Business logic, no HTTP knowledge
- `src/db/`          Knex models and migrations
- `src/lib/`         Shared utilities

## Don't
- Don't use `any` in TypeScript.
- Don't query the DB from inside route handlers — go through services.
- Don't add new dependencies without asking.

演習2（中級）— 「Conventional Commits」Skill を作る

所要時間: 20 分 / 到達点: /conventional-commit を実行すると Claude が staged diff を読んで適切なメッセージを提案する。

  1. .claude/skills/conventional-commit/SKILL.md を作成。
  2. frontmatter に name, description, disable-model-invocation: true, allowed-tools を設定。
  3. SKILL.md の本文に「①staged 差分を見る ②type と scope を決める ③subject と body を書く ④ユーザに確認 ⑤commit を実行」の 5 ステップを書く。
  4. 適当な変更を git add し、/conventional-commit で動作確認。
📖 参考解答

「2.4 ハンズオン」セクションの SKILL.md がそのまま解答になります。実行時のチェックポイント:

  • /skills で一覧に表示されるか。
  • disable-model-invocation: true なので Claude が勝手に発動しないこと（手動 /conventional-commit のみで動く）。
  • commit 前に必ず提案メッセージを表示して、ユーザの承認を待つこと。

演習3（上級）— レガシーモジュール背景知識 Skill

所要時間: 30 分 / 到達点: src/legacy/** を編集している時だけ自動有効化される、ユーザには出てこない背景知識 Skill を作る。

  1. .claude/skills/legacy-context/SKILL.md を作成。
  2. frontmatter で次を指定:
    • name: legacy-context
    • description: 何の知識か + いつトリガーされるかを記述
    • paths: ["src/legacy/**"]
    • user-invocable: false（ユーザがスラッシュメニューから呼べない）
  3. 本文に、レガシーモジュールの設計判断・落とし穴・代替案を 300 行以内で記述。
  4. 長い詳細は reference.md に分離して SKILL.md から参照。
  5. src/legacy/foo.ts を Claude に編集してもらい、Skill が機能するか確認。

# Legacy billing module

The `src/legacy/` directory contains the original billing implementation
written before we adopted Prisma. It uses a custom data-mapper pattern.

## Hard rules
- Do not introduce Prisma here. Use the existing `LegacyRepo` base class.
- Do not change the table schema; many internal scripts depend on it.
- Add tests under `tests/legacy/` only.

## Gotchas
- `Invoice.total` is stored in cents but exposed as a float. See [reference.md](reference.md) for the conversion history.
- The `audit_log` table is append-only. Updates require a new row with `revision++`.

## Migration plan (in progress)
- See @docs/legacy-migration.md for the rollout phases.

📖 用語集（Glossary）

CLAUDE.md
Claude Code がセッション開始時に自動で読み込む Markdown ファイル。プロジェクトの長期記憶として機能する。
CLAUDE.local.md
個人ローカル用の CLAUDE.md。Git にはコミットしない。CLAUDE.md と並んでロードされる。
Auto memory
Claude 自身が会話の中から学んだ内容を ~/.claude/projects/<project>/memory/ 配下に書き溜める仕組み。
Skill (Agent Skill)
特定タスクの手順・知識・スクリプトを 1 ディレクトリにまとめた拡張機能。Claude が必要時にオンデマンドで読み込む。
SKILL.md
Skill ディレクトリのエントリポイントとなる Markdown ファイル。YAML frontmatter に name と description を持つ。
Progressive disclosure
必要な情報を必要な時にだけロードする方式。Skill は metadata（常時）→ SKILL.md 本文（発火時）→ 参照ファイル/スクリプト（必要時）の 3 段階。
サブエージェント (Subagent)
独立コンテキストで動く別の AI 作業者。.claude/agents/ や ~/.claude/agents/ に Markdown で定義する。
スラッシュコマンド
/init, /memory, /skills, /compact など、Claude Code で / を頭に付けて呼び出す内蔵機能。Skill も /skill-name で呼べる。
Hooks
ツール呼び出しの前後やセッションイベントに紐付けて実行されるシェルコマンド。「必ず実行されるべき」処理を決定論的に強制する。
YAML frontmatter
Markdown ファイルの先頭に --- で囲んで書く YAML メタデータ。Skill ではここで name, description 等を宣言する。
.claude/rules/
大規模プロジェクト向けの、CLAUDE.md を複数ファイルに分割するための仕組み。各ファイルに paths を指定するとパス特化型ルールになる。
Fork (subagent context)
Skill の frontmatter に context: fork を指定すると、その Skill は独立コンテキスト（サブエージェント）で実行される。大量のデータを扱うのに有効。
/init
リポジトリを解析して CLAUDE.md の雛形を自動生成する Claude Code コマンド。
プラグイン (Plugin)
Skill / サブエージェント / コマンドなどをまとめて配布する単位。/plugin marketplace add <repo> で外部から導入可能。
