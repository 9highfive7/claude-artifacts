Claude Code「CLAUDE.md」と Agent Skills 実践ガイド

Claude Codeをチームで使うときに重要な CLAUDE.md と Agent Skills を、 概念・構文・実装・運用まで一気通貫で学ぶ教材です。

最終更新: 2026-05-22 対象: 開発者 / Tech Lead / DevEx 学習目安: 3〜5時間 ハンズオン: 60〜90分
検証条件: このHTMLは公式ドキュメントURLを参照元として整理していますが、本生成環境ではWeb取得・CLI実測を実行していません。 CLI出力、Skills API、CLAUDE.local.md の扱いなどはバージョン依存のため、実運用前に公式リンクで再確認してください。
検索条件に一致するセクションがありません。

このガイドの読み方

学習目標

  • 初級者・経験者それぞれの最短ルートを選べる。
  • 公式情報と運用例の区別を意識して読む。

本ガイドは「概念 → 構文 → 実践 → 運用」の順に進みます。 まずCLAUDE.mdでClaude Codeに常時渡す文脈を整え、次にSkillsでタスク固有の専門能力をパッケージ化します。

1. 概念何を解決する仕組みか

第1部: CLAUDE.md 編

学習目標
  • CLAUDE.mdの役割、読込階層、書き方を説明できる。
  • プロジェクトに適したCLAUDE.mdテンプレートを作れる。
  • 悪い指示を、Claude Codeが実行しやすい指示に改善できる。

1. CLAUDE.mdとは何か

公式 主出典: Claude Code Memory docs

CLAUDE.md は、Claude Codeに渡す長期記憶・プロジェクトコンテキストをMarkdownで記述するファイルです。 毎回チャットで「このプロジェクトはpnpmを使います」「テストはこのコマンドです」と説明する代わりに、 リポジトリ内の知識として固定できます。

役割: CLAUDE.mdは「Claudeに常に覚えておいてほしい、リポジトリ固有の情報」を置く場所です。 汎用的なプログラミング解説ではなく、あなたのプロジェクト固有の事実・規約・禁止事項を書きます。
  • 解決する課題: 文脈説明の重複、コマンド誤実行、規約忘れ、オンボーディングのばらつき。
  • 読まれるタイミング: Claude Codeのセッション開始時や、関連ディレクトリを扱うタイミング。
  • 注意点: 常時コンテキストに入るため、冗長にするとトークンを消費します。

2. 配置場所と階層

代表的な配置は次のとおりです。細かな探索順・自動読込仕様はバージョン依存のため、 最新のMemory docsで確認してください。

場所用途Git管理注意
./CLAUDE.md プロジェクト共有メモリ。規約・コマンド・設計概要。 通常コミットする チーム全員に有効な情報だけを書く。
~/.claude/CLAUDE.md 個人のグローバルメモリ。好みの応答形式など。 通常コミットしない チーム規約を書かない。
./path/CLAUDE.md サブディレクトリ固有の文脈。例: frontend/, infra/。 内容次第でコミット 必要時に読み込まれる設計として使う。
CLAUDE.local.md 個人設定・ローカルパス・実験的指示。 コミットしない 要確認 自動読込可否は最新仕様を確認。確実にするなら@CLAUDE.local.mdで明示インポート。
@path/to/file CLAUDE.mdから別ファイルを取り込む。 参照先次第 長い設計資料・コマンド一覧を分割するのに便利。

3. 書き方の基本

CLAUDE.mdはMarkdownで書きます。見出しを分け、Claudeがすぐ使える「事実」と「行動指示」を短く書きます。 

# Project: <project-name>

## Overview
- このプロジェクトは <何をするシステムか>。
- 主要技術: <例: TypeScript, Next.js, PostgreSQL>。

## Commands
- `pnpm install`: 依存関係をインストール。
- `pnpm dev`: ローカル開発サーバー。
- `pnpm test`: テスト実行。
- `pnpm lint`: lint実行。
- `pnpm typecheck`: 型チェック。

## Code Style
- TypeScriptはstrict前提。
- `any` は原則禁止。必要な場合は理由をコメントに書く。
- APIクライアントは `src/lib/api/` に置く。

## Testing
- 変更した機能にはユニットテストを追加する。
- E2Eは重いため、明示依頼がない限り関連ケースだけ実行する。

## Do Not
- `generated/` 配下を手編集しない。
- マイグレーションを勝手に削除しない。
- 秘密情報をコードやCLAUDE.mdに書かない。
目安: まずは「概要」「コマンド」「規約」「テスト」「禁止事項」の5セクションから始めると、過不足が少なくなります。 100〜200行を超える詳細は、@docs/... に分割するのが運用しやすいです。 運用例

4. ベストプラクティス

DODON'T
プロジェクト固有のコマンドを書く。 「きれいなコードを書いて」のような一般論だけを書く。
命名規則・ディレクトリ規約・禁止事項を書く。 古い設計メモや変更頻度の高いTODOを大量に書く。
Claudeに実行してほしい検証手順を書く。 実在しないコマンドや古いコマンドを放置する。
/initで初期生成し、PRで育てる。 個人の秘密情報やアクセストークンを書く。

理解度チェック: CLAUDE.md

Q1. CLAUDE.mdに「一般的な良いコードの書き方」を長く書くべきでない理由は？

毎回コンテキストを消費するためです。CLAUDE.mdにはプロジェクト固有のコマンド、規約、禁止事項を短く書くのが効果的です。

Q2. チーム共有の規約と個人のローカルパスはどこに分けるべき？

共有規約は ./CLAUDE.md、個人情報はGit管理外の CLAUDE.local.md やユーザーメモリに分けます。

Q3. 長い設計資料をCLAUDE.mdに直接貼らずに扱う方法は？

@docs/architecture.md のようにインポートで分割します。詳細な制限は最新公式docsを確認してください。

この章の要点: CLAUDE.mdは「常時必要なプロジェクト文脈」の置き場です。短く、具体的に、チーム共有できる内容を書き、個人情報や機密情報は含めません。

第2部: Skills（Agent Skills）編

学習目標
  • SkillsをCLAUDE.md・サブエージェント・スラッシュコマンドと区別できる。
  • SKILL.md と補助ファイルを含むSkillを作成できる。
  • description設計とProgressive disclosureの重要性を説明できる。

1. Skillsとは何か

公式 主出典: Agent Skills overview / getting started / best practices

Agent Skills は、特定タスクの手順・知識・スクリプト・テンプレートを1つのディレクトリにまとめ、 Claudeが必要時に呼び出せるようにする拡張パッケージです。

メンタルモデル: CLAUDE.mdが「常に持つプロジェクト記憶」なら、Skillは「必要なときだけ取り出す専門能力」です。 たとえば「社内のリリースノート形式で書く」「Conventional Commitsに従ってコミットを作る」 「ブランドガイドに沿って資料を作る」といったタスクに向きます。
仕組み読み込まれ方向いている用途
CLAUDE.md 常時または階層に応じて読み込まれる プロジェクト全体の規約・コマンド・設計概要
Skills descriptionを手がかりにオンデマンドで読み込まれる 特定タスクの手順、テンプレート、補助スクリプト
Subagents 別の専門エージェントとして委任 別コンテキストでの調査、レビュー、監査
Slash commands ユーザーが明示的に実行 定型プロンプトやワークフローのショートカット

2. 構造とファイル形式

最小構成は、Skill名のディレクトリと SKILL.md です。 SKILL.md のYAMLフロントマターには、少なくとも name と description を書きます。 

.claude/
└── skills/
    └── conventional-commit/
        ├── SKILL.md
        ├── scripts/
        │   └── suggest_commit.py
        └── templates/
            └── commit-message.txt
---
name: conventional-commit
description: Use when creating, reviewing, or improving git commit messages that must follow Conventional Commits. Trigger on requests about git commit, staged changes, commit message, feat/fix/chore/docs/test/refactor, or release history.
---

# Conventional Commit

Use this skill to help create safe, clear commit messages.

## Workflow
1. Inspect staged changes with `git diff --cached --stat` and `git diff --cached`.
2. Identify the most appropriate type: feat, fix, docs, test, refactor, chore, perf, build, ci.
3. Propose one concise subject line.
4. Do not run `git commit` unless the user explicitly confirms.
descriptionが重要: Claudeは最初にSkillのnameとdescriptionを見て、使うべきか判断します。 「何をするSkillか」だけでなく、「どんな依頼文・文脈で使うか」を具体的に書きます。

補助ファイルには、スクリプト、テンプレート、チェックリスト、参考資料を入れられます。 Claudeに毎回手で判断させるより、機械的にできる検証はスクリプト化すると安定します。

3. 動作の仕組み: Progressive disclosure

Skillsの重要概念が Progressive disclosure です。 すべてのSkill本文を最初から読むのではなく、まずdescriptionだけを見て、必要と判断したSkillの詳細を読みます。

  • 自動選択: ユーザー依頼とdescriptionが合うと、ClaudeがSkillを使う。
  • 明示呼び出し: 「conventional-commit skillを使って」のように依頼できる。
  • 失敗例: descriptionが曖昧、複数Skillが重複、必要なファイルが足りない。

4. ハンズオン: Conventional Commit Skillを作る

Step 1: ディレクトリを作る

mkdir -p .claude/skills/conventional-commit/scripts
mkdir -p .claude/skills/conventional-commit/templates

Step 2: SKILL.mdを書く

---
name: conventional-commit
description: Use when creating or reviewing git commit messages from staged changes. This skill helps write Conventional Commits such as feat, fix, docs, test, refactor, chore, build, ci, perf. Trigger on git commit, commit message, staged diff, release notes, or change summary.
---

# Conventional Commit Skill

## Goal
Create a clear, safe commit message that follows Conventional Commits.

## Rules
- Format: `type(scope): subject`
- Subject must be imperative, concise, and under 72 characters when practical.
- Allowed types: feat, fix, docs, test, refactor, chore, build, ci, perf.
- If the change is ambiguous, ask the user before committing.
- Never run `git commit` unless the user explicitly confirms.

## Workflow
1. Check staged files:
   - `git diff --cached --name-status`
   - `git diff --cached --stat`
2. Run `scripts/suggest_commit.py` if available.
3. Inspect the diff when necessary.
4. Propose 1 primary message and up to 2 alternatives.
5. Ask for confirmation before executing `git commit`.

## Output format
- Suggested: `type(scope): subject`
- Why: one short sentence.
- Risk: mention if staged changes look too broad.

Step 3: 補助スクリプトを入れる

#!/usr/bin/env python3
"""
Suggest a Conventional Commit skeleton from staged files.

This script is intentionally conservative:
- It does not create commits.
- It only reads staged file names and diff stats.
"""

from __future__ import annotations

import os
import subprocess
import sys
from collections import Counter
from pathlib import Path


def run_git(args: list[str]) -> str:
    try:
        result = subprocess.run(
            ["git", *args],
            check=True,
            text=True,
            stdout=subprocess.PIPE,
            stderr=subprocess.PIPE,
        )
        return result.stdout.strip()
    except subprocess.CalledProcessError as exc:
        print(exc.stderr.strip() or str(exc), file=sys.stderr)
        sys.exit(exc.returncode)


def staged_files() -> list[str]:
    output = run_git(["diff", "--cached", "--name-only"])
    return [line.strip() for line in output.splitlines() if line.strip()]


def infer_type(files: list[str]) -> str:
    if not files:
        return "chore"
    if all(f.endswith((".md", ".mdx", ".rst")) or f.startswith("docs/") for f in files):
        return "docs"
    if all("/test" in f or f.endswith(("_test.py", ".test.ts", ".spec.ts")) for f in files):
        return "test"
    if any(f.startswith((".github/", "ci/")) for f in files):
        return "ci"
    if any(Path(f).name in {"package.json", "pnpm-lock.yaml", "package-lock.json", "uv.lock"} for f in files):
        return "build"
    return "chore"


def infer_scope(files: list[str]) -> str:
    top_levels = [Path(f).parts[0] for f in files if Path(f).parts]
    if not top_levels:
        return "repo"
    [(scope, _count)] = Counter(top_levels).most_common(1)
    return scope.replace("_", "-")


def main() -> None:
    files = staged_files()
    if not files:
        print("No staged files. Stage changes first, e.g. `git add <files>`.")
        sys.exit(1)

    commit_type = infer_type(files)
    scope = infer_scope(files)
    print(f"Suggested skeleton: {commit_type}({scope}): <imperative summary>")
    print()
    print("Staged files:")
    for file in files:
        print(f"- {file}")
    print()
    print("Tip: inspect `git diff --cached` and replace the summary before committing.")


if __name__ == "__main__":
    main()

5. Skillsのベストプラクティス

  • descriptionは具体的に: タスク名、関連キーワード、使うべき状況を書く。
  • 1 Skill = 1責務: 「リリース」「コミット」「DB migration」など境界を明確にする。
  • スクリプトを活用: ファイル一覧、lint、変換など機械的処理はスクリプト化する。
  • 失敗時の手順を書く: 権限不足、ファイル不在、曖昧な入力のときにClaudeが迷わないようにする。
  • レビューする: SkillはClaudeの行動を変えるため、チーム共有する場合はコードレビュー対象にする。

理解度チェック: Skills

Q1. Skillのdescriptionが重要な理由は？

ClaudeがSkillを使うかどうかを最初に判断する手がかりだからです。タスク名だけでなく、発火してほしい文脈やキーワードを書く必要があります。

Q2. CLAUDE.mdではなくSkillにすべき情報は？

常時必要ではなく、特定タスクでだけ必要な手順・テンプレート・スクリプトです。例: リリースノート生成、コミットメッセージ作成。

Q3. Skillにスクリプトを入れる利点は？

ファイル一覧取得、形式変換、検証などの決定的処理を安定化でき、Claudeの推測依存を減らせます。

この章の要点: Skillsはオンデマンドで読む専門能力です。description設計、責務の分離、補助スクリプト、安全な実行手順が品質を左右します。

第3部: 比較と使い分け

学習目標
  • CLAUDE.md / Skills / Subagents / Slash commandsを用途で選べる。
  • 複数の仕組みを組み合わせた運用パターンを設計できる。
  • 「何でもCLAUDE.mdに書く」状態を避けられる。

比較表

仕組み主な目的起動方法保存場所例向くもの避けるもの
CLAUDE.md 常時必要な記憶 セッション開始・階層に応じて読込 ./CLAUDE.md, ~/.claude/CLAUDE.md コマンド、規約、設計概要、禁止事項 長大な手順書、個人秘密情報、頻繁に変わるTODO
Skills タスク固有の専門能力 descriptionに基づき自動または明示使用 .claude/skills/<name>/SKILL.md リリース、コミット、帳票、社内形式、補助スクリプト 常に必要な基本規約、曖昧で巨大な万能Skill
Subagents 別役割への委任 自動選択または明示依頼 .claude/agents/*.md レビュー、調査、セキュリティ監査、専門担当 単純な定型プロンプト
Slash commands 手動実行する定型操作 /command .claude/commands/*.md PRレビュー依頼、要約、定型ワークフロー 自動発火してほしい専門手順

判断フロー

  1. その情報は、ほぼすべての作業で必要か？ → CLAUDE.md
  2. 特定タスクのときだけ必要な手順・資料・スクリプトか？ → Skill
  3. 別の専門家として独立した文脈で調査・レビューさせたいか？ → Subagent
  4. 人間が明示的に呼び出すショートカットか？ → Slash command
  5. 権限、hooks、環境設定を制御したいか？ → Settings

組み合わせ運用パターン

パターン構成例
最小チーム標準 CLAUDE.md + settings コマンド、規約、禁止事項、許可ツールをチームで共有。
定型業務自動化 CLAUDE.md + Skills CLAUDE.mdに全体規約、Skillにリリースノートやコミット手順。
レビュー強化 Slash command + Subagent + Skill /review-pr でレビュー担当subagentを呼び、必要時にセキュリティSkillを参照。

理解度チェック: 使い分け

Q1. 「このリポジトリではpnpmを使う」はどこに書く？

ほぼ全作業で必要なので CLAUDE.md が適切です。

Q2. 「障害報告書を社内テンプレートで作る」はどれが適切？

特定タスクの手順とテンプレートなのでSkillが適切です。

Q3. 「セキュリティ専門家として別観点でレビュー」は？

Subagentが向いています。必要に応じてセキュリティSkillを併用します。

第4部: 運用とチーム展開

学習目標
  • Git管理すべきファイルと除外すべきファイルを分けられる。
  • CLAUDE.mdとSkillsのレビュー・更新サイクルを設計できる。
  • セキュリティ上の注意点をチーム標準に落とし込める。

1. Gitで管理するもの / しないもの

コミット推奨コミット非推奨
  • CLAUDE.md
  • .claude/skills/** のチーム共有Skill
  • .claude/commands/** の共有コマンド
  • .claude/agents/** の共有subagent
  • .claude/settings.json はレビュー前提で検討
  • CLAUDE.local.md
  • .claude/settings.local.json
  • 個人トークン、秘密鍵、社外秘データ
  • ~/.claude/ 配下の個人設定
  • Claudeの会話ログや一時ファイル

2. レビュー・更新サイクル

  • PR必須: CLAUDE.mdや共有Skillの変更はコード同様にレビューする。
  • オーナーを決める: DevEx、Tech Lead、各ドメイン担当が承認する。
  • 四半期棚卸し: 存在しないコマンド、古い設計、使われないSkillを削除する。
  • 変更理由を書く: 「なぜこの指示を足したか」をPR本文に残す。
  • オンボーディングに組み込む: 新メンバーはCLAUDE.mdと主要Skillを読む。

3. セキュリティ上の注意

絶対に避ける: APIキー、秘密鍵、顧客データ、未公開脆弱性、個人情報をCLAUDE.mdやSkillに直接書かないでください。 ローカル専用でも、共有・誤コミット・ログ出力のリスクがあります。
  • 破壊的操作を行うSkillには「明示確認なしに実行しない」と書く。
  • 補助スクリプトは通常のコードと同じくレビュー・テストする。
  • 外部通信、ファイル削除、DB migration、deployは権限と承認フローを明確にする。
  • 社内規約や法務・コンプライアンス要件は最新の内部資料にリンクする。

理解度チェック: 運用

Q1. 共有Skillをレビュー対象にすべき理由は？

SkillはClaudeの行動を変え、スクリプト実行や生成物品質に影響するためです。通常のコードと同様に安全性・正確性を確認します。

Q2. 個人のローカルパスや好みはどこに置く？

Git管理外の CLAUDE.local.md、.claude/settings.local.json、またはユーザーグローバルメモリに置きます。

Q3. 古くなったCLAUDE.mdを放置すると何が起きる？

Claudeが存在しないコマンドを実行したり、古い設計前提で修正したりして、作業品質が下がります。

学習目標

  • 小さなSkillを作って動作確認できる。

演習1: 自分のリポジトリ用CLAUDE.mdを作る

以下を含む30〜80行のCLAUDE.mdを作ってください。

  • プロジェクト概要
  • 開発・テスト・lint・型チェックコマンド
  • ディレクトリ構成
  • 禁止事項
  • 変更後の確認手順
解答例を表示 
# My API

## Overview
- 社内向け在庫API。
- Stack: Python 3.12, FastAPI, PostgreSQL, pytest.

## Commands
- `uv run pytest`: tests.
- `uv run ruff check .`: lint.
- `uv run mypy app`: type check.
- `docker compose up db`: local database.

## Architecture
- `app/routes/`: HTTP routes.
- `app/services/`: business logic.
- `app/repositories/`: database access.
- `migrations/`: Alembic migrations.

## Rules
- routeから直接SQLを書かない。
- DB変更時はmigrationを追加する。
- PIIをfixtureに入れない。

## Done Criteria
- 関連テスト、ruff、mypyを実行し、結果を回答に含める。

演習2: 「リリースノート生成」Skillを設計する

.claude/skills/release-note-writer/SKILL.md を作り、descriptionとワークフローを書いてください。

解答例を表示 
---
name: release-note-writer
description: Use when writing release notes from git log, pull requests, issue lists, or changelog drafts. Trigger on release note, changelog, version summary, shipped changes, bug fixes, breaking changes.
---

# Release Note Writer

## Goal
Create release notes in the team's standard format.

## Inputs
- version
- date
- git range or PR list
- audience: internal, customer, developer

## Workflow
1. Collect changes from the provided range or PR list.
2. Group into Added, Changed, Fixed, Deprecated, Removed, Security.
3. Highlight breaking changes.
4. Do not invent issue numbers or customer impact.
5. Ask for missing version/date if not provided.

## Output
- Title: `vX.Y.Z - YYYY-MM-DD`
- Summary: 2-3 sentences.
- Sections: Added / Changed / Fixed / Breaking Changes.

演習3: どの仕組みを使うか選ぶ

  1. 全開発者が守るReactコンポーネント規約。
  2. DB migrationの危険操作をレビューする専門担当。
  3. 毎週、同じ形式で週報を作るショートカット。
  4. 顧客向けリリースノートをテンプレートに沿って生成。
解答例を表示
  1. CLAUDE.md。常時必要なチーム規約だから。
  2. Subagent。専門ロールで別観点レビューしたいから。必要に応じてSkillも併用。
  3. Slash command。人間が明示実行する定型プロンプトだから。
  4. Skill。特定タスクの手順・テンプレート・分類ルールが必要だから。

用語集（Glossary）

CLAUDE.md
Claude Codeにプロジェクト固有の長期記憶・指示を渡すMarkdownファイル。
Agent Skill / Skill
特定タスクの手順、知識、スクリプト、テンプレートをパッケージ化した拡張機能。
SKILL.md
Skillの中心ファイル。YAMLフロントマターに name と description を持つ。
Progressive disclosure
最初は軽量な説明だけを読み、必要になったら詳細を読む設計。Skillsのコンテキスト効率を支える考え方。
Subagent
特定ロール・ツール・文脈を持つ専門エージェント。レビュー、調査、監査などに向く。
Slash command
/init やカスタム /review-pr のように、ユーザーが明示的に呼び出すコマンド。
Front matter
Markdown冒頭の --- で囲まれたYAMLメタデータ。
Context window
モデルが一度に参照できる入力範囲。CLAUDE.mdを長くしすぎると消費が増える。
@ import
CLAUDE.md内で @docs/file.md のように別ファイルを取り込む機能。
