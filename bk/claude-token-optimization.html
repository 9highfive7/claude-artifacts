はじめに

モデルの品質は落とさず、トークン消費だけを減らす。このガイドは、Claude Code および Claude API を利用する開発者のための、トークンコスト最適化テクニック集です。「安いモデルに切り替える」以外のすべての手法を網羅しています。

企業の平均利用コストは1日あたり約$13/開発者（月額$150-250）ですが、適切な最適化で50〜90%のコスト削減が可能です。

ⓘ 対象バージョン

Claude Code CLI v2.1.121 / Claude API (2026年5月時点) に基づいています。価格や機能は変更される可能性があります。

最終更新: 2026-05-22

⚡ クイックウィン TOP5 — まずこれをやれ

1

タスク完了ごとに /clear する（効果: 即座にコスト低減）

Claude はメッセージごとに全会話履歴を再読み込みします。50発目のメッセージは5発目より大幅にコストが高くなります。タスクが変わったら /clear で新しいセッションを開始しましょう。「クリアしない方がもったいない」が正解です。

2

フェーズ完了ごとに /compact する（効果: 80-90%圧縮）

20,000+トークンの会話履歴を2,000-3,000トークンに圧縮します。劣化を感じてからでは遅い — 調査フェーズ完了時、実装完了時など、フェーズの切れ目で実行するのが最適です。

3

CLAUDE.md を200行以下に保つ（効果: ベースラインコスト削減）

CLAUDE.md は毎リクエストに読み込まれます。5,000トークンの CLAUDE.md は、何もしなくても毎回コストが発生します。冗長な情報を削り、タスク固有の指示はスキルに移動しましょう。

4

プロンプトキャッシュを活用する（効果: 入力コスト90%削減）

API利用時、キャッシュ読み込みは基本入力価格の0.1倍。システムプロンプトに cache_control を付けるだけで、繰り返し呼び出しのコストが劇的に下がります。

5

不要なMCPサーバー・ツールを無効化する（効果: コンテキスト15-30%削減）

有効な全ツールの定義がコンテキストを消費します。使わないMCPサーバーは外し、--allowedTools で必要なツールだけ有効化しましょう。

A. コンテキスト管理

/clear で会話履歴をリセット

簡単 効果: 大

タスクが完了したら /clear で新しいセッションを開始します。古いデバッグログや試行錯誤の履歴が残ると、以降のすべてのメッセージでコストが増加します。

/clear
💡 ヒント

「クリアするともったいない」は誤解です。クリアしない方がはるかにコストがかかります。

/compact で会話履歴を要約圧縮

簡単 効果: 大

20,000+トークンの会話を2,000-3,000トークンに圧縮します。カスタム指示で保持する情報を制御できます。

/compact                              # デフォルト圧縮
/compact Focus on code samples and API usage  # 保持情報を指定

いつ実行するか: 調査→実装、実装→テストなどフェーズが変わるタイミングが最適。コンテキスト警告が出てからでは遅い場合があります。

セッションを短く保つ

簡単 効果: 中

Claude はメッセージごとに全会話履歴を再読み込みします。メッセージ50は、メッセージ5と比べて蓄積されたコンテキスト分だけコストが高くなります。1セッション1トピックの原則を意識しましょう。

.claudeignore / settings.json で読み込み除外

簡単 効果: 中

node_modules、dist、build、ログファイルなど不要なディレクトリをブロックします。曖昧なプロンプトは大規模なコードベーススキャンを誘発します。

# .claudeignore の例
node_modules/
dist/
build/
*.log
coverage/
.next/

B. プロンプトキャッシュの活用

ⓘ 対象

プロンプトキャッシュは Claude API を直接利用する場合のテクニックです。Claude Code CLI を通常利用する場合は自動的に管理されます。

cache_control の基本

中級 効果: 大

システムプロンプトやツール定義に cache_control を付与すると、以降のリクエストでキャッシュから読み込まれます。キャッシュ読み込みは入力価格の0.1倍（90%割引）。

response = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "あなたは経験豊富なソフトウェアエンジニアです...",
            "cache_control": {"type": "ephemeral"}  # 5分TTL
        }
    ],
    messages=[{"role": "user", "content": "このコードをレビューして"}],
)
操作コスト倍率Opus 4.7 (per MTok)
通常入力1.0x$5.00
キャッシュ書き込み (5m)1.25x$6.25
キャッシュ書き込み (1h)2.0x$10.00
キャッシュ読み込み0.1x$0.50

キャッシュヒット率を最大化する

上級 効果: 大

最重要テクニック: 静的コンテンツを先頭、動的コンテンツを末尾に配置。ProjectDiscovery社はこの再配置だけでキャッシュ率を7%→84%に改善し、コスト59%削減を達成しました。

3段ブレークポイント構成

system=[
    # BP1: 静的システムプロンプト（1h TTL = 長期キャッシュ）
    {"type": "text", "text": "静的な指示...",
     "cache_control": {"type": "ephemeral", "ttl": "1h"}},

    # BP2: ツール定義（静的なものから順に配置）
    {"type": "text", "text": "ツール定義...",
     "cache_control": {"type": "ephemeral"}},

    # BP3: 会話コンテキスト（5m TTL = 頻繁に変わる部分）
    {"type": "text", "text": "最近の会話...",
     "cache_control": {"type": "ephemeral", "ttl": "5m"}}
]
⚠ キャッシュを殺す行為

システムプロンプトへのタイムスタンプ埋め込み、空白のドリフト（\n\n と \n の混在）、頻繁なスキーマバージョン変更 — これらはキャッシュを無効化します。

キャッシュの損益分岐点と制約

中級 効果: 中

5分以内に同一セグメントで1.25回以上呼び出せばペイします。本番環境では容易に超えます。

モデル最小キャッシュ長
Opus 4.7/4.6/4.5, Haiku 4.54,096 トークン
Sonnet 4.6/4.5, Opus 4.11,024 トークン

最大4つのブレークポイント/リクエスト。キャッシュは最初のレスポンス開始後に利用可能になります。

キャッシュのプリウォーム

上級 効果: 小

最初のリクエストのレイテンシを避けるため、max_tokens: 0 でキャッシュだけ作成できます。

prewarm = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=0,  # 出力なし（キャッシュ作成のみ）
    system=[{
        "type": "text",
        "text": "大量のシステムプロンプト...",
        "cache_control": {"type": "ephemeral"}
    }],
    messages=[{"role": "user", "content": "warmup"}],
)

C. CLAUDE.md / メモリの最適化

CLAUDE.md を200行以下に保つ

簡単 効果: 大

CLAUDE.md は毎リクエストに読み込まれます。1,000行の CLAUDE.md を80行に削減した事例もあります。すべてのセッションで必要な情報だけを残しましょう。

💡 実践ルール

「このセッションでも次のセッションでも必要か？」を基準に取捨選択。タスク固有の指示はスキル (.claude/skills/) に移動しましょう。

メモリの階層化とスコープ制御

中級 効果: 中

MEMORY.md は先頭200行 or 25KB のみロードされます。path-scoped rules (.claude/rules/) を使えば、特定のファイルパターンにマッチした場合のみ指示が読み込まれます。

# .claude/rules/frontend.md (*.tsx にのみ適用)
---
paths: ["src/components/**/*.tsx"]
---
React コンポーネントは関数コンポーネントで書く。
CSS-in-JS は使わず Tailwind を使用する。

指示をスキルに移動する

中級 効果: 中

CLAUDE.md に全指示を詰め込む代わりに、タスク固有の指示をスキルとして分離します。スキルは呼び出し時のみロードされるため、約30%のトークン節約になります。

D. ツール使用・MCP の最適化

不要な MCP サーバー・ツールを無効化

簡単 効果: 中

有効な全ツールの定義テキストがコンテキストを消費します。使わない MCP サーバーは外し、必要なツールだけ有効化しましょう。

# 必要なツールだけ指定して起動
claude --allowedTools "Bash(git *) Edit Read Glob Grep"

# MCP サーバーを削除
claude mcp remove unused-server

ツール出力の蓄積を抑える

中級 効果: 大

ファイル読み込み、MCP レスポンス、シェルコマンド出力はすべてコンテキストに永続的に追加されます。これが最大のサイレントドレインです。出力サイズに上限を設けましょう。

# settings.json の env で出力上限を設定
{
  "env": {
    "BASH_MAX_OUTPUT_LENGTH": "20000",
    "MAX_MCP_OUTPUT_TOKENS": "8000"
  }
}

スキル一覧のバジェットを制限

中級 効果: 小

設定でスキル一覧に割り当てるコンテキスト量を制御できます。

{
  "skillListingBudgetFraction": 0.01,
  "maxSkillDescriptionChars": 1536
}

E. プロンプト設計の工夫

明確・簡潔な指示

簡単 効果: 中

冗長な前置きを削り、具体的に指示します。「何をどうしてほしいか」を直接的に書くことで、入力トークンと出力トークンの両方を削減できます。

# NG: 冗長
「もしよろしければ、このファイルの中で、パフォーマンスに関連する
問題がないかどうか、確認していただけると助かります。」

# OK: 簡潔
「src/api.ts のパフォーマンス問題を指摘して。」

例示（few-shot）を最適化する

中級 効果: 中

3-5個の例が最も効果的です。10個以上の例を入れてもほとんど精度は上がらず、トークンだけ消費します。例は多様性・関連性を重視しましょう。

拡張思考のコントロール

中級 効果: 大

思考トークンは最大のコストドライバーです。Effort Level を下げるか、必要時以外は無効化します。30-40%のコスト削減が期待できます。

# Claude Code 内でエフォートレベルを変更
/effort low       # 短いタスク向け
/effort medium    # コスト重視
/effort high      # 推奨バランス（デフォルト）

# API での Adaptive Thinking
client.messages.create(
    model="claude-opus-4-7",
    max_tokens=64000,
    thinking={"type": "adaptive"},
    output_config={"effort": "high"},
    messages=[...]
)
Effort Level用途コスト影響
max最大能力が必要な場合最高コスト（収穫逓減あり）
xhighコーディング・エージェント作業高
highほとんどのタスクの推奨値中〜高
mediumコスト重視のタスク中
low短い質問・レイテンシ重視低（30-40%削減）

出力スタイルの制御

簡単 効果: 小

Markdown の箇条書きは散文より冗長になりがちです。公式ドキュメントでは、長文出力には prose（散文）を推奨しています。

F. ファイル読み込みの工夫

必要な範囲だけ読み込む

簡単 効果: 中

大きなファイルは全文ではなく、offset と limit を指定して必要な範囲だけ読みましょう。API の Read ツールは行番号指定に対応しています。

# Claude Code での指示例
「src/app.ts の 100-150 行目を読んで」
「package.json の dependencies セクションだけ見て」

grep / glob で事前に絞り込む

簡単 効果: 中

ファイルを直接読む前に、grep や glob で対象を絞り込みます。バイナリファイルや巨大ログファイルを誤って読ませると大量のトークンが浪費されます。

# Claude Code への指示例

  見つかったファイルの該当箇所だけ読んで」

バッチ + キャッシュの組み合わせ

上級 効果: 大

H. 出力量のコントロール

max_tokens の適切な設定

簡単 効果: 中

API 利用時、max_tokens を不必要に大きく設定すると、Claude が長文を生成しがちです。タスクに応じた適切な上限を設定しましょう。

# 短い応答が欲しい場合
client.messages.create(
    model="claude-opus-4-7",
    max_tokens=256,  # 必要十分な値に制限
    messages=[...]
)

出力の簡潔さを指示する

簡単 効果: 小

「簡潔に」「3行以内で」「JSON のみ出力」などの明示的な指示で、不要に長い応答を防ぎます。最初に要件を明確にすることで、やり直し（＝追加トークン）も減らせます。

I. モニタリング

/usage でセッションコストを確認

簡単 効果: 小

Claude Code 内で /usage を実行すると、現在のセッションのトークン使用量とコストを確認できます。最適化の効果測定に活用しましょう。

/usage

ccusage でローカルコスト追跡

簡単 効果: 中

ccusage（GitHub 4,800+ stars）は、Claude Code のローカルセッションファイルを読み取ってコストを可視化するツールです。API キー不要でプライバシーに配慮されています。

npx ccusage claude daily     # 日次レポート
npx ccusage monthly          # 月次レポート
npx ccusage session          # セッション別コスト
bunx ccusage blocks          # 5時間課金ウィンドウ分析

Token Counting API で事前見積もり

中級 効果: 小

API のトークンカウントエンドポイントを使えば、実際にリクエストを送る前にトークン数を見積もれます。無料で利用可能（レート制限あり）。

curl https://api.anthropic.com/v1/messages/count_tokens \
  --header "x-api-key: $ANTHROPIC_API_KEY" \
  --header "content-type: application/json" \
  --header "anthropic-version: 2023-06-01" \
  --data '{
    "model": "claude-opus-4-7",
    "messages": [{"role": "user", "content": "テスト"}]
  }'

Anthropic Console ダッシュボード

簡単 効果: 小

Anthropic Console の Usage ページで、ワークスペース全体の使用量とコストを確認できます。予算上限の設定も可能です。

J. アーキテクチャレベルの工夫

サブエージェントでコンテキストを隔離

中級 効果: 大

テスト実行、ドキュメント取得、ログ処理などの冗長な出力を生むタスクはサブエージェントに委任します。冗長出力はサブエージェントのコンテキスト内に留まり、メイン会話には要約だけが返ります。

# Claude Code 内での指示例
「テストの実行とログ分析はサブエージェントに任せて、
  結果のサマリーだけ報告して」
⚠ 注意: Agent Teams

Agent Teams（複数エージェントの協調）は通常セッションの約7倍のトークンを消費します。コスト意識が必要な場面では単一エージェント+サブエージェントの構成を検討しましょう。

大タスクを小タスクに分割

簡単 効果: 中

探索フェーズと実装フェーズを分離し、各フェーズを独立したセッションで実行します。探索の結果を CLAUDE.md やスキルに記録してから実装に入ると、コンテキストの無駄な蓄積を防げます。

フックで前処理する

上級 効果: 中

フック（PreToolUse、PostToolUse）を使って、Claude に渡す前にデータをフィルタリング・加工できます。大量のログやAPI応答を事前に要約してからコンテキストに追加する使い方が有効です。

FAQ

/clear と /compact、どちらを使うべき？

/clear は完全にセッションをリセットし、新しい会話を開始します。タスクが完全に切り替わる場合に使います。/compact は同じタスク内で会話を圧縮し、重要な情報を保持したまま続行します。同じタスクの途中（フェーズの区切り）では /compact、別タスクに移る時は /clear が最適です。

プロンプトキャッシュは Claude Code でも使えるの？

Claude Code CLI は内部的にプロンプトキャッシュを自動管理しています。ユーザーが明示的に cache_control を設定する必要はありません。プロンプトキャッシュのテクニック（静的コンテンツを先頭に配置するなど）は、Claude API を直接利用するアプリケーション開発で特に重要です。

Effort Level を下げると品質は落ちる？

low にすると思考トークンが大幅に減り、複雑な推論が必要なタスクでは品質に影響します。ただし、「ファイルの修正箇所を教えて」「このエラーの意味は？」のような単純なタスクでは low でも十分な品質です。タスクの複雑さに合わせて使い分けましょう。high がほとんどのタスクでの推奨値です。

バッチ処理とキャッシュは同時に使える？

はい、割引はスタック（重複適用）します。バッチ処理の50%割引とキャッシュ読み込みの90%割引が組み合わさると、キャッシュされた入力トークンは最大95%のコスト削減になります。バッチ内のキャッシュ TTL は 1h を推奨します。

ccusage はどうやってコストを計算している？

ccusage はローカルの Claude Code セッションファイル（JSONL 形式）を読み取り、各メッセージのトークン数と利用モデルから料金を計算します。API キーは不要で、すべてローカルで完結します。npx ccusage claude daily で日次レポートを確認できます。
