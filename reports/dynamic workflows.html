★概要 ── 30秒で要点

  • Dynamic workflow とは: Claude が書いた JavaScript スクリプトが、ループ・分岐・中間結果を自分で保持しながら、最大数百のサブエージェントを動かす仕組み。デフォルトのコーディング向けハーネスとは別に、タスク専用のハーネスをその場で生成できる。
  • なぜ重要か: 単一コンテキストで長時間・大規模・敵対的なタスクを進めると、agentic laziness・self-preferential bias・goal drift という失敗モードが起きやすい。別々のコンテキストを持つ Claude に分割することで構造的に防げる。
  • すぐ試す: /deep-research <質問>。自分のタスクは ultracode キーワードか「ワークフローを使って」で。
  • 非技術タスクにも強い: 履歴書ランキング、事業計画の多角レビュー、Slack のインシデント根本原因調査など。
  • 注意: トークン消費が大きい。まず小さな範囲で試し、必要なら 「10k tokens まで」のような予算指定を。
本ガイドは公式リファレンスに加え、Anthropic の「A harness for every task」記事の実践的知見を統合したものです。ベストプラクティスは現在も発展途上です。

1なぜ dynamic workflows なのか

デフォルトのハーネスは、計画と実行を同じコンテキストウィンドウで行います。多くのコーディングタスクでは有効ですが、長時間・大規模並列・高度に構造化された敵対的タスクでは破綻することがあります。

同一コンテキストで複雑なタスクに長く取り組むほど、Claude は次の特定の失敗モードに陥りやすくなります。ワークフローは、それぞれ独立したコンテキストウィンドウと焦点を絞った隔離されたゴールを持つ複数の Claude をオーケストレーションすることで、これらに対抗します。

失敗モード説明ワークフローによる対策
Agentic laziness
（怠惰）		 複雑で多部構成のタスクで、部分的な進捗のまま完了を宣言してしまう。例: セキュリティレビューの50項目中20項目だけ対処して終わる。 各項目を独立エージェントに割り当て、スクリプトが完了をすべて待つ（barrier）。
Self-preferential bias
（自己優先バイアス）		 自分の結果や発見を、特にルーブリックで検証・採点させたときに優先しがち。 別の独立エージェントが敵対的に検証する（adversarial verification）。
Goal drift
（目標のずれ）		 多数のターン、特に compaction（要約圧縮）後に、当初の目的への忠実度が徐々に失われる。「Xをするな」のような制約やエッジケース要件が欠落しやすい。 各エージェントに焦点を絞った固定ゴールを与え、スクリプトが原典の目標を保持する。

2dynamic と static の違い

以前から Claude Agent SDK や claude -p で複数の Claude Code を協調させる static workflow を作れました。違いは「事前に汎用的に書く」か「その場で専用に書かせる」かです。

Static workflowDynamic workflow
作る人人間が事前にコード化（SDK / claude -p）Claude がその場でタスク専用に生成
汎用性全エッジケース対応のため汎用的・冗長になりがちユースケースに合わせ tailor-made
前提—Claude Opus 4.8 級の知能で初めて実用的に
調整コードを書き直すプロンプトでナッジ、保存後に編集
workflow は数十行の特殊関数で subagent を spawn・coordinate する JavaScript ファイルです。JSON・Math・Array など標準関数も使え、各エージェントが使うモデルや worktree で隔離するかをスクリプトが決められます。中断されても、セッション再開で続きから再開します。

5設計パターン 6種

Claude がワークフロー構築時に使い、組み合わせる代表的なパターンです。カードを選ぶと、下のビルダーがプロンプトの叩き台を組み立てます。

✓Classify-and-act

分類エージェントがタスク種別を判定し、種別ごとに別エージェント／挙動へルーティング。最後に分類で出力を決めることも。

✓Fan-out-and-synthesize

タスクを多数の小ステップに分割し、各ステップを別エージェントで実行。すべて待ってから（barrier）構造化出力を1つに統合。

✓Adversarial verification

生成エージェントごとに、別の検証エージェントを立て、ルーブリックや基準に対して敵対的に出力を検証する。

✓Generate-and-filter

多数のアイデアを生成し、ルーブリックや検証でフィルタ、重複を除去し、最高品質で検証済みのものだけを返す。

✓Tournament

分割せず競争させる。N体が異なるアプローチで同じタスクに挑み、判定エージェントがペアワイズに比較して勝者を決める。

✓Loop until done

作業量が未知のタスクで、停止条件（新たな発見なし／ログにエラーなし）を満たすまでエージェントを spawn し続ける。

6ユースケース集

技術・非技術を問わず、ワークフローが効く具体的な場面です。

領域やり方の要点主なパターン
マイグレーション／リファクタcallsite・失敗テスト・モジュール等の単位に分解。修正ごとに worktree でサブエージェントを起動→別エージェントが敵対的レビュー→マージ。並列を最大化するため、エージェントに高負荷コマンドを避けるよう指示すると安全。（Bun の Zig→Rust 書き換えで使用）fan-out / adversarial
Deep research/deep-research は Web 検索をファンアウト→ソース取得→主張を敵対的検証→引用付きレポートを合成。Web に限らず Slack の状況報告や、コードベース深掘りでの「機能の仕組み」調査にも応用可。fan-out / adversarial
Deep verification1エージェントが全ての事実主張を抽出し、主張ごとにサブエージェントを立てて詳細検証。さらに検証エージェントがソースの質を点検。ブログ草稿の技術的主張をコードベースに照合する等。fan-out / adversarial
ソート1000行超を1プロンプトで並べると品質劣化＆コンテキスト超過。代わりにトーナメント（ペアワイズ比較は絶対採点より信頼性が高い）やバケットランクを並列実行してマージ。決定論的ループがブラケットを保持し、走査順だけがコンテキストに残る。tournament
メモリ／ルール遵守CLAUDE.md に入れても守られないルールは、ルール1つにつき検証エージェント1体で点検。誤検知を減らすため懐疑役（skeptic）ペルソナでルールを精査。逆向きに、過去セッションやレビューコメントから繰り返す修正を抽出→クラスタリング→敵対的検証→CLAUDE.md に蒸留も有効。fan-out / adversarial
根本原因調査独立した複数仮説を立てて検証するのがデバッグの定石。logs・files・data など互いに素な証拠から仮説を生成するエージェントを立て、各仮説を検証者・反証者のパネルにかける。コードに限らず、売上分析やデータパイプラインの post-mortem にも。generate-and-filter / adversarial
大規模トリアージ各項目を分類→既存追跡分とdedupe→アクション（修正試行 or 人へエスカレーション）。quarantine パターン: 信頼できない公開コンテンツを読むエージェントには高権限アクションを禁じ、行動は別エージェントが担う。/loop と組み合わせて継続実行。classify / loop
探索とテイストデザインや命名などテイストに依る課題で、多数の案を探索しレビューエージェントにルーブリックを与える。基準を満たした時点で完了。トーナメントで順位付け・選定も。generate-and-filter / tournament
Evalsworktree で別エージェントを起動し、比較エージェントがルーブリックに対して採点。作成した skill を特定基準で評価・改善する等の軽量 eval に。tournament / adversarial
モデル／知能ルーティングタスクに合わせた分類エージェントが、事前リサーチで最適モデルを判定。例: 「auth モジュールの仕組み説明」は対象ファイル数と構造に応じて Sonnet か Opus にルーティング。classify

7構築のTips

プロンプティング

上記パターンの用語を使った詳細なプロンプトが最良の結果を生みます。ワークフローは大きなタスク専用ではなく、「quick workflow」も指定できます（例: ある前提の素早い敵対的レビュー）。

/goal と /loop を組み合わせる

トリアージ・リサーチ・検証など繰り返せるワークフローは、/loop で定期実行し、/goal で「これを満たすまで止まるな」という強い完了条件を設定すると効果的です。

トークン予算

「use 10k tokens」 のように明示すると上限が設定され、暴走を抑えられます。

保存と共有

ワークフローメニューで s を押して保存し、~/.claude/workflows にチェックインするか、skill として配布できます。skill 経由で共有するには、JavaScript ファイルを skill フォルダに置き SKILL.md から参照します。柔軟性のため、skill 内のワークフローは「逐語実行するスクリプト」ではなくテンプレートとして扱うよう Claude に促すとよいでしょう。

8使わない方がよい時

すべてのタスクに必要なわけではありません。 ワークフローは大きな成果を生む場面が多い一方、不要なタスクでは大幅にトークンを浪費します。通常のコーディングタスクでは「本当に追加の compute が必要か？」を自問してください。多くの伝統的なコーディングタスクに、5人のレビュアーのパネルは要りません。

ワークフローは、これまでできなかった方向に Claude Code を押し広げるために創造的に使うのが最良です。

9同梱ワークフローを実行する

最短で動きを確認するなら /deep-research。質問を多角的に調べ、ソースを相互検証して引用付きレポートを返します。

  1. ワークフローを実行する調べたい質問とともに実行。複数角度に Web 検索をファンアウトし、ソースを取得・相互検証し、引用付きレポートを合成します。 
    /deep-research What changed in the Node.js permission model between v20 and v22?
  2. ワークフローを許可するClaude Code が可否を尋ねます。Yes を選択（プロンプトは権限モードによる、承認参照）。
  3. 進捗を見るバックグラウンドで実行され、セッションは空いたまま。/workflows で実行を選び Enter で進捗ビュー（フェーズごとのエージェント数・トークン・経過時間）。 
    /workflows
  4. レポートを読む完了するとレポートが届きます。出典が引用され、相互検証を生き残らなかった主張は除外済みです。

10Claude にワークフローを書かせる

プロンプトで依頼する

effort を変えずに単一タスクをワークフロー化するには ultracode を含めます。「ワークフローを使って」など自然言語でも同じオプトインです。

ultracode: audit every API endpoint under src/routes/ for missing auth checks
バージョン差分: v2.1.160 より前のリテラルトリガーは workflow。自然言語の依頼は両バージョンで動作。誤起動は macOS Option+W / Win・Linux Alt+W で解除、または /config でトリガーをオフ。

ultracode で Claude に任せる

xhigh reasoning effort と自動オーケストレーションを組み合わせた設定。実質的な各タスクごとに Claude がワークフローを計画します。

/effort ultracode
現在のセッションのみ有効でリセットされます。通常作業は /effort high に戻します。xhigh 対応モデルでのみ利用可。

実行前にプランを承認する

CLI のプロンプトは計画フェーズと選択肢（Yes, run it / Yes, and don't ask again… / View raw script / No）を表示。Ctrl+G でエディタに開き、Tab でプロンプト調整。

権限モードプロンプトが出るタイミング
Default / accept edits毎回（「don't ask again」選択時を除く）
Auto初回のみ。以後は同意を記録し無確認。ultracode 時はスキップ
Bypass / claude -p / Agent SDK尋ねず即開始
重要: 生成されるサブエージェントは常に acceptEdits で動作し、ツール許可リストを継承。allowlist 外のシェル・Web フェッチ・MCP は途中で確認することあり。長い実行では事前に allowlist へ追加を。

保存して再利用する

/workflows で実行を選び s。Tab で保存先を切り替え（.claude/workflows/=プロジェクト共有 / ~/.claude/workflows/=自分専用）。Enter で保存し /<名前> として実行。同名はプロジェクト側が優先。

入力を渡す（args）

保存ワークフローは args グローバルで入力を受け取れます。構造化データで渡されるため配列・オブジェクトメソッドを直接呼べます。省略時は undefined。

> Run /triage-issues on issues 1024, 1025, and 1030

11実行の仕組みと制限

ランタイムは会話と別の隔離環境でスクリプトを実行。中間結果はスクリプト変数に残ります。

各実行はスクリプトを ~/.claude/projects/ 配下に書き出し、開始時に Claude がパスを受け取ります。読んだり、過去実行と diff したり、編集して再起動を頼めます。ランタイムが各エージェントの結果を追跡するため、同一セッション内で再開可能です。

制約理由
実行中のユーザー入力なし停止できるのはエージェントの権限プロンプトのみ。段階承認が必要なら各段を別ワークフローに
ワークフロー自体からの FS／シェル直接アクセスなし読み書き・実行はエージェント。スクリプトは調整役
同時実行 最大16エージェント（低コア機ではより少）ローカルリソース抑制
1実行あたり合計1,000エージェント暴走ループ防止

12実行の管理とコスト

一時停止後の再開

停止後も再開可能。完了済みはキャッシュ結果を返し、残りはライブ実行。/workflows で選び p、または再起動を Claude に依頼。同一セッション内のみで、Claude Code 終了後の次セッションは最初からになります。

コスト

  • まず小さく試す: 1ディレクトリ／狭い質問で。/workflows が各エージェントのトークンを表示、完了分を失わず停止可。
  • 上限が歯止め: エージェント上限が暴走コストを抑制。
  • モデル確認: 大きな実行前に /model。不要なステージは小さいモデルを Claude に指定。予算は 「10k tokens まで」のように指定。

無効化

自分用は /config のトグル、~/.claude/settings.json の "disableWorkflows": true、または CLAUDE_CODE_DISABLE_WORKFLOWS=1。組織用は managed settings か管理設定ページのトグル。無効時は同梱コマンド不可、ultracode 起動なし、/effort から ultracode が消えます。
